## 1. Feasibility - 実現可能性

計画が技術的に実現可能で、妥当なコストで実装できるか。

### チェック項目

- [ ] 影響ファイルが実在し、指定行番号が正しいか（コードを実際に確認）
- [ ] 使用するAPI/ライブラリが実在し、計画通りのインターフェースを持つか
- [ ] 実装環境の制約（ランタイム制限、プラットフォーム互換性等）を満たすか
- [ ] 推定工数が妥当か（過少/過大見積もりがないか）
- [ ] 依存関係が正しい順序で実装可能か
- [ ] 既存テストへの影響範囲が考慮されているか

### 信頼スコア基準

| スコア | 状態 |
|--------|------|
| 80-100 | 実在しないAPIの使用、不可能な実装方法、致命的な環境制約の見落とし |
| 50-79 | 行番号のズレ、工数の大幅なずれ、不明確な実装手順 |
| 0-49 | 軽微な指摘のみ、実現可能 |

## 2. Security - セキュリティ

入力検証・機密データ保護・インジェクション等のセキュリティリスク。

### チェック項目

- [ ] 外部からの入力（ユーザー入力、ネットワーク、ファイル）が全て検証・サニタイズされるか
- [ ] 機密データ（認証情報、トークン、鍵等）が安全に扱われるか（ログ出力、平文保存の禁止）
- [ ] インジェクション攻撃（SQL、コマンド、パス等）に対する防御があるか
- [ ] 外部リソースへのアクセス時にリダイレクト・SSRF等のリスクが考慮されているか
- [ ] エラーメッセージにシステム内部情報が漏洩しないか
- [ ] 信頼境界をまたぐデータフローが安全か

### 信頼スコア基準

| スコア | 状態 |
|--------|------|
| 80-100 | 未検証の外部入力が危険な操作に渡される、機密データの露出 |
| 50-79 | 検証はあるが不完全、新規入力パスのテスト不足 |
| 0-49 | セキュリティ対策が適切 |

## 3. Performance & Memory - パフォーマンス・メモリ効率

CPU/メモリの効率的な使用、スケーラビリティ。

### チェック項目

- [ ] O(n^2)以上のアルゴリズムが大量データで使われていないか
- [ ] リソースリーク（ファイルハンドル未クローズ、リスナー未解放、タイマー未クリア等）がないか
- [ ] 不要なコピーやアロケーション（ループ内の文字列結合、不要なディープコピー等）がないか
- [ ] 大きなデータのメモリ保持期間が最小化されているか
- [ ] 非同期処理の適切な使用（不要な直列化、並列化可能な処理の未活用）
- [ ] 重複計算・重複I/O・重複処理がないか
- [ ] ランタイム固有のリソース制約（スレッド数、メモリ上限等）を考慮しているか

### 信頼スコア基準

| スコア | 状態 |
|--------|------|
| 80-100 | リソースリーク確実、O(n^2)で大量データ処理、メモリ枯渇リスク |
| 50-79 | 非効率だが機能する、改善の余地あり |
| 0-49 | 効率的な実装 |

## 4. Architecture & Design - アーキテクチャ・設計妥当性

プロジェクトのアーキテクチャ原則と設計方針への適合。

### チェック項目

**プロジェクト固有（CLAUDE.md / .claude/review-rules.md から読み取る）:**
- [ ] CLAUDE.md に定義されたレイヤー構造への違反がないか
- [ ] CLAUDE.md に定義された依存方向ルールへの違反がないか
- [ ] `.claude/review-rules.md` に定義されたプロジェクト固有の設計ルール違反がないか

**汎用:**
- [ ] DRY原則: 2回以上の重複コードが存在しないか
- [ ] 単一責任: 1ファイル/1関数が複数の責務を持っていないか
- [ ] 型安全: 適切な型定義があるか、型の緩い使い方をしていないか
- [ ] エラーハンドリング: 適切な伝播とリカバリーがあるか

### 信頼スコア基準

| スコア | 状態 |
|--------|------|
| 80-100 | プロジェクト定義のレイヤー違反、禁止された依存方向 |
| 50-79 | DRY違反、不適切な責務分離、型安全性の不足 |
| 0-49 | アーキテクチャ準拠 |

## 5. Completeness - 考慮漏れ・網羅性

計画に含まれるべき事項が漏れていないか。

### チェック項目

- [ ] エラーハンドリング: 全ての失敗パスで適切なフォールバックがあるか
- [ ] エッジケース: 空入力、巨大入力、不正入力、Unicode、マルチバイト文字
- [ ] 後方互換性: 既存の設定/データ/APIとの互換性が保たれるか
- [ ] テスト計画: 各変更に対応するテストが計画されているか
- [ ] ロールバック可能性: 問題発生時に安全に戻せるか
- [ ] リソースクリーンアップ: 確保したリソースの解放が保証されるか
- [ ] ドキュメント更新: 設定ファイル、API、アーキテクチャドキュメント等の更新が必要か

### 信頼スコア基準

| スコア | 状態 |
|--------|------|
| 80-100 | テスト計画なし、致命的エッジケースの見落とし、エラーハンドリング欠如 |
| 50-79 | 一部のエッジケース漏れ、ドキュメント更新漏れ |
| 0-49 | 十分に網羅的 |

## 6. Alternatives - 代替手法の検討

提案された実装方法よりも優れた代替手法がないか。

### チェック項目

- [ ] 同じ目的を達成するよりシンプルな方法がないか
- [ ] 言語・ランタイムの標準ライブラリで代替できないか（車輪の再発明防止）
- [ ] 既存のライブラリ/ユーティリティで代替できないか
- [ ] 将来の拡張性を考慮した設計になっているか
- [ ] パフォーマンスとコード量のトレードオフは妥当か

### 信頼スコア基準

| スコア | 状態 |
|--------|------|
| 80-100 | 標準ライブラリで簡潔に解決できるのに複雑な実装を計画している |
| 50-79 | より良い方法があるが、計画の方法でも機能する |
| 0-49 | 最適なアプローチ |
