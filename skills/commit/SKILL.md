# Auto Commit

ユーザーに確認を求めず、変更を分析して論理単位で自動コミットするスキル。

## 基本原則

1. **確認しない** — 分析→実行を一気通貫で行う。ユーザーへの確認プロンプトは出さない
2. **論理単位** — 関連する変更を1つのコミットにまとめ、無関係な変更は分割する
3. **中途半端な状態でコミットしない** — 壊れた状態のコードをコミットしない
4. **安全第一** — 破壊的操作は絶対に行わない

## Phase 1: 情報収集

以下を**並列実行**する：

```bash
git status
git diff
git diff --staged
git log -5 --format='[%h] %an <%ae> | %ar | %s'
git log -1 --format='%H %an %ae'
git branch -vv
```

## Phase 2: 健全性チェック

以下のいずれかに該当する場合は**コミットを中止**し、理由を報告する：

### 中止条件

- **変更がない**: staged も unstaged もない場合
- **main/master ブランチ**: 直接コミットしない（ユーザーに警告して終了）
- **マージコンフリクト中**: 未解決のコンフリクトマーカーがある場合
- **明らかに壊れた状態**:
  - 構文エラーが含まれるファイル（変更対象が少数の場合のみ簡易チェック）
  - import/require先が存在しないモジュール参照の追加

> **注意**: 健全性チェックは軽量に行う。ビルドやテストの全実行はしない。明らかな構文破壊のみ検出する。

## Phase 3: 分析と戦略決定

### 3.1 変更の分類

各ファイルの変更を以下の軸で分類する：

- **パス**: src/ / test/ / docs/ / config/ 等
- **変更の性質**: feat / fix / refactor / docs / test / chore / style / perf
- **関連性**: 同じ機能・目的に属するか

### 3.2 コミット戦略

#### A. 単一コミット

すべての変更が同じ目的・テーマに属する場合。

```
例: src/auth.ts, src/auth.test.ts, src/types/auth.ts
→ すべて認証機能に関連 → 1つのコミット
```

#### B. 複数コミットに分割

独立した変更が混在する場合、論理単位で分割する。

```
例:
  変更1: src/user.ts (新機能)
  変更2: README.md (ドキュメント更新)
→ feat: ... と docs: ... の2コミットに分割
```

**分割の判断基準:**
- 異なる Conventional Commits タイプが混在（feat + docs 等）
- 独立した機能/バグ修正が複数ある
- 将来別々に revert する可能性がある

**分割しない場合:**
- テストと実装が同じ機能に属する（feat にまとめる）
- 型定義の追加と、それを使う実装コード

#### C. Amend

以下の**すべて**を満たす場合のみ、直前コミットに amend する：

1. 直前のコミットが未 push（リモート追跡ブランチより ahead）
2. 直前のコミットの作成者が自分
3. 変更内容が直前のコミットと明確に同じテーマ
4. 統合後のサイズが妥当（50ファイル未満 & 1000行未満）
5. 統合後も単一の論理単位として理解可能

## Phase 4: 実行

### コミットメッセージ形式

```
<type>: <subject>

<body（必要な場合のみ）>
```

- **type**: Conventional Commits（feat / fix / docs / style / refactor / perf / test / chore）
- **subject**: 日本語または英語（プロジェクトの既存コミット履歴のスタイルに合わせる）
- **body**: 変更の背景説明が必要な場合のみ。不要なら省略
- **フッター**: デフォルトでは付けない

### 実行手順

1. 戦略に従い `git add` で必要なファイルをステージング
   - `git add -A` や `git add .` は使わない。ファイルを個別指定する
   - `.env`、credentials、秘密鍵等のセンシティブファイルは除外する
2. `git commit` を実行（HEREDOC形式でメッセージを渡す）
3. 複数コミットの場合は順番に繰り返す
4. amend の場合は `git commit --amend` を使用

### Pre-commit Hook 失敗時

1. Hook が自動修正したファイルを確認
2. 自動フォーマット等の安全な修正の場合：
   - 修正されたファイルを再ステージング
   - **新規コミットとして**再実行（amend しない — hook 失敗時はコミット自体が作成されていないため）
3. それ以外のエラー：理由を報告して中止

## Phase 5: 報告

実行結果を簡潔に報告する。

### 成功時

```
✅ <コミット数>件コミット完了

[abc1234] feat: ユーザー認証機能を追加
  - src/auth.ts, src/auth.test.ts, src/types/auth.ts

[def5678] docs: README を更新
  - README.md
```

### 中止時

```
⛔ コミット中止: <理由>
```

## 禁止事項

- `--no-verify` の使用
- push 済みコミットへの amend
- `git rebase -i` の自動実行
- `git push --force`
- `git reset --hard`
- ユーザーへの確認プロンプト（AskUserQuestion の使用）
