# Planning Discussion Flow

チーム計画作成の議論フローの詳細手順。

## 概要

4段階の議論プロセスを経て、多角的な視点を持つ実装計画を作成する。

```
Step 1: 要件共有（Lead → 全員） → ユーザーの要件 + コードベース情報を全メンバーに共有
Step 2: 設計提案（全員 → Lead）  → 各自の専門観点から設計ポイント・懸念を提出
Step 3: 議論（全員）            → トレードオフについて多角的に議論する
Step 4: 計画作成（Lead）        → 合意に基づいて計画ファイルを作成する
```

## Step 1: 要件共有（Lead → 全員）

### 目的

ユーザーの要件とコードベースの関連情報を全メンバーに共有し、議論の土台を作る。

### 手順

1. Lead がユーザーの要件を整理する
2. コードベースを調査し、関連ファイルの構造・既存実装を把握する
3. 以下の情報を SendMessage で全メンバーに共有する:
   - ユーザーの要件（何を作りたいのか）
   - コードベースの関連情報（既存の類似実装、影響範囲）
   - プロジェクトの制約・ルール（CLAUDE.md の内容）

### SendMessage テンプレート（全員宛）

```
## 計画作成: 要件共有

### ユーザーの要件
{requirements}

### コードベースの関連情報
{codebase_context}

### プロジェクトルール
{claude_md_content}

上記を踏まえ、あなたの専門観点から設計提案・懸念事項を提出してください。
```

## Step 2: 設計提案（全員 → Lead、並行）

### 目的

各メンバーが自分の専門観点から、設計上考慮すべきポイントを提案する。

### 手順

1. 各メンバーが Lead からの要件共有を受け取る
2. 自分の専門観点から以下を検討する:
   - 設計上考慮すべきポイント
   - やるべきこと / やらないべきこと
   - 懸念事項
3. SendMessage で Lead に報告する

### 各ロールの提案フォーカス

| ロール | フォーカス |
|--------|-----------|
| **Security Advisor** | セキュリティ要件、入力検証、機密データ保護、最小権限 |
| **Performance Advisor** | 効率的なデータ構造・アルゴリズム、リソース管理、並列化 |
| **Architect** | レイヤー構造、依存方向、テスタビリティ、拡張性 |
| **Pragmatist** | スコープの適切さ、YAGNI、既存ライブラリの活用、実装順序 |

### 提案フォーマット

```
### {ロール名} 設計提案

**推奨事項:**
- {設計で考慮すべきポイント}

**やるべきこと:**
- {含めるべき要素}

**やらないべきこと:**
- {スコープ外にすべき要素}

**懸念事項:**
- {リスクや注意点}
```

## Step 3: 議論（全員、最大3ラウンド）

### 目的

トレードオフが発生する論点について、各メンバーが自分の立場から意見を述べ、最適解を見つける。

### 手順

1. Lead が各メンバーの提案を整理し、以下に分類する:
   - **合意事項**: 複数メンバーが同意する設計ポイント（議論不要）
   - **トレードオフ論点**: メンバー間で意見が対立する論点（議論対象）
2. トレードオフ論点がなければ Step 4 にスキップ
3. トレードオフ論点を全メンバーに共有（SendMessage）
4. 各メンバーの意見を SendMessage で受け取る
5. **各ラウンド終了時にサマリーを出力する**（後述のフォーマット参照）
6. **早期収束条件を確認する**: 全論点が WARN 以下に収束していたら残りのラウンドをスキップ
7. 最大 3 ラウンドで収束させる
8. 合意に至らない場合は Lead が判断（理由を明記）

### 議論のルール

- 各論点について **最大 3 ラウンド** の往復で収束させる（[共通原則](../../shared/references/team-config.md)参照）
- **早期収束条件**: ラウンド終了時に全論点が WARN 以下に収束していたら残りのラウンドをスキップ（重大度定義は [severity-and-verdicts.md](../../shared/references/severity-and-verdicts.md) 参照）
- 合意に至らない場合は Lead が判断する（理由を明記）
- 「批判」ではなく「提案」の姿勢を維持する

### 典型的なトレードオフ

- **Security vs Simplicity**: セキュリティ層の追加 vs シンプルな実装
- **Architecture の理想 vs Pragmatist の現実**: 拡張性のための抽象化 vs 工数削減
- **Performance vs Readability**: 最適化 vs コードの可読性
- **Scope**: 含めるべき機能の範囲

### 議論ラウンドの進め方

```
ラウンド 1:
  Lead → 全員: 論点提示
  全員 → Lead: 各自の意見（SendMessage）
  Lead: ラウンドサマリー出力 + 早期収束チェック

ラウンド 2（必要な場合のみ）:
  Lead → 全員: 反論 or 妥協案の提示
  全員 → Lead: 最終意見（SendMessage）
  Lead: ラウンドサマリー出力 + 早期収束チェック

ラウンド 3（必要な場合のみ）:
  Lead → 全員: 最終確認
  全員 → Lead: 最終意見（SendMessage）
  Lead: ラウンドサマリー出力
```

### ラウンドサマリー出力フォーマット

Lead が各ラウンド終了時にユーザー向けに以下を出力する:

```
── Round {N}/{max} ──
論点: {topic}
  Security: {stance_summary} [BLOCK/WARN/INFO/PASS]
  Performance: {stance_summary} [BLOCK/WARN/INFO/PASS]
  Architect: {stance_summary} [BLOCK/WARN/INFO/PASS]
  Pragmatist: {stance_summary} [BLOCK/WARN/INFO/PASS]
  Status: {resolved | ongoing}
```

## Step 4: 計画作成（Lead）

### 目的

議論の結果を踏まえて、合意に基づく実装計画を作成する。

### 手順

1. 合意事項と議論結果を整理する
2. 計画ファイルを作成する:
   - `skills/plan/references/plan-template.md` のフォーマットに準拠
   - 議論のハイライトを計画ファイルに含める
   - 各メンバーの提案がどう反映されたかを記録
3. `docs/cycles/{timestamp}_{slug}.md` に出力する

### 計画ファイルに含める追加セクション

通常の計画テンプレートに加え、以下のセクションを含める:

```markdown
## Team Planning Results

**Planned:** {datetime}
**Team:** Security Advisor, Performance Advisor, Architect, Pragmatist

### 合意事項
- {合意した設計ポイント}

### 議論ハイライト
- {論点}: {合意内容の要約}

### 各メンバーの貢献
- **Security Advisor**: {主な貢献}
- **Performance Advisor**: {主な貢献}
- **Architect**: {主な貢献}
- **Pragmatist**: {主な貢献}
```

## Step 5: メタレビュー（条件付き）

### 目的

議論で BLOCK が発生した場合に、修正の妥当性とクロスカッティングリスクをチェックする。

### 発火条件

- 議論の合意形成時に **BLOCK が1つでもあった場合のみ** 自動発火
- 全論点が WARN 以下で収束した場合はスキップ

### 手順

メタレビューの条件判定・ラウンド制限・入力制限のルールは [severity-and-verdicts.md](../../shared/references/severity-and-verdicts.md) の「メタレビュー規則」に準拠する。

1. 合意結果の修正差分 + 合意サマリーを作成する
2. **BLOCK/WARN を出したメンバーのみ** に修正確認を SendMessage で依頼する（全員に再送しない）
3. 各メンバーの確認結果を受け取る
4. クロスカッティングリスクを明示的にチェック:
   - 複数提案の組み合わせで発生するリスク
   - 修正が他の合意事項と矛盾していないか
5. Lead が最終判断を下す

### 制約

- **最大1回**: メタレビューで新たな BLOCK が出ても Lead が最終判断を下す
- **入力は差分のみ**: 計画全文の再送は禁止（トークン節約）
- **対象メンバー限定**: BLOCK/WARN を出したメンバーのみ

## エラーケース

### メンバー spawn 失敗

- 成功したメンバーが 2 名以上: 残りのメンバーで議論を続行
- 成功したメンバーが 1 名以下: TeamDelete して中断

### メンバーからの提案がない

- 提案がないメンバーは議論から除外
- 2 名以上の提案があれば続行

### 議論が収束しない

- 最大 3 ラウンドで打ち切り（[共通原則](../../shared/references/team-config.md)参照）
- Lead が最終判断を下す（理由を明記）
