# Session workflow

session は対話を入口、台帳の状態遷移を出口にする。人間が正本を読んでいない前提で、状況と推奨を平易に提示する。沈黙、推奨への無反応、LLM の解釈は承認ではない。

## 1. Prepare the session

その場記録モードはリスク順、考古学モードは物語順で提示する。考古学モードでは個別裁定の前に orient と現状仕様リファレンスで文脈を回復する。

冒頭に対象テーマ、裁定件数、進め方、中断可能であることを3〜5行で説明する。判断根拠が同じ行を一つのテーマにまとめ、次の未充足スロットだけを人へ聞く。

- 目的: 利用者が観測する振る舞い
- 採否: 採用または却下
- 適用範囲
- 例外
- 委任条件

回答は `OK` / `違う` / `任せる` / `保留` の4択。高リスク行には既定回答を置かない。`違う` の理由は同じ回答イベントへ保存し、代案は新しい未裁定claimとして扱う。`保留` は状態を変えない。

## 2. Interpret before recording

対象行を名指しした明示的な直接回答だけは、そのまま記録段階へ進める。複数行に跨る回答、条件付き回答、曖昧な態度、指示語、または `任せる` の対象が一意でない回答には解釈確認ゲートを使う。

確認前に提示できるのは、各行についての「解釈案」と確認質問だけである。この段階では次を生成・表示しない。

- 状態遷移、遷移矢印、確定後の状態
- approval object、digest、batch manifest
- 「記録する」「合意済みにする」など、すでに確定したように読める表現
- `AGREED` / `REJECTED` / `DELEGATED` を値にした予定レコード

代わりに「UI-042 は OK という理解で合っていますか」のように、候補の意味を自然言語で示す。曖昧な指示語や `任せる` があれば、何を指すかが曖昧であることを明記してから解釈案を出す。人間が解釈案を明示確認するまで全対象行を現状態のまま保つ。

`任せる` を含む場合は、主体、対象操作、scope（既定は現 plan）、期限、取消を含む最小委任案を一度提示し、どの行または判断部分への委任かを確認する。

## 3. Record only after confirmation

人間が解釈案を確認した後に初めて、確認済み回答を session 成果物へ記録する。成果物は `schema_version`、`session_id`、`responses[{row_id, revision, answer}]`、必要なら `reason` と `batch_summary` を持つ。確認待ちの回答にも、次工程が「人間の確認 → session 成果物の生成 → `ledger_write` で記録」であることを一文で示す。

その成果物を `ledger_write.py` の `approve` / `reject` / `batch-approve` に渡す。CLI は承認を代行せず、確認済みの人間回答を記録するだけである。digest や approval object を手書きしない。書き込み時にだけ [agreement-ledger.md](../../shared/references/agreement-ledger.md) の構造契約を読み、記録後は `ledger_lint.py` で検証する。

テーマ末尾では新しい合意間と既存合意との矛盾を確認する。高リスク行と異論のあった行はbatchへ混ぜない。疲労を検知したら中断し、残件を未裁定のままにして再開地点を示す。

## Completion

実行した書き込み・lintコマンド、exit code、検出件数、状態変更、未解決・保留を報告する。人間の確認がまだなら、確認待ちであり状態変更0件と明示する。
