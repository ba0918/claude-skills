# ledger 提示テンプレート集

合意台帳（[agreement-ledger.md](../../shared/references/agreement-ledger.md)）の各ワークフローが使う
テンプレート。台帳ファイル・CONTEXT 語彙ファイルは LLM 向け（機械検証正本）、裁定行提示・status 裁定ビューは
**人間向け**である。テンプレート内の値はすべて合成・匿名データであり、機密情報を書かない。

## 状態語 → 人間向けラベル対応表

status 裁定ビューには**生の状態語を出さない**。次の対応で変換する:

| 状態語（台帳・LLM 向け） | 人間向けラベル |
|--------------------------|---------------|
| `AGREED` | 合意済み |
| `DELEGATED` | 任せた（範囲: …） |
| `PROVISIONAL` | 暫定（再評価: …） |
| `UNDECIDED` | 未裁定 |
| `REJECTED` | 却下 |

## 台帳ファイルテンプレート（JSON・LLM 向け）

トップレベルは `schema_version` と `rows`。各行は発話サイズの主張 1 つ。extract は全行を `UNDECIDED` /
`PROVISIONAL` で生成する（`AGREED` は session の人間承認からのみ）。

```json
{
  "schema_version": 1,
  "rows": [
    {
      "id": "NAV-001",
      "revision": 1,
      "state": "UNDECIDED",
      "claim": "トップナビはグローバル固定で全画面共通とする",
      "term_refs": ["T-NAV"],
      "observations": ["既存モックにヘッダー領域がある"],
      "assumptions": ["全画面でナビが必要と仮定"]
    },
    {
      "id": "AUTH-010",
      "revision": 2,
      "state": "AGREED",
      "claim": "ログインは合成 SSO のみを受け付ける",
      "approval": {
        "row_id": "AUTH-010",
        "revision": 2,
        "digest": "<claim + term_refs から再算出される digest>",
        "session_id": "S-20260721-01",
        "actor_kind": "human",
        "prior_state": "UNDECIDED"
      }
    },
    {
      "id": "IMPL-020",
      "revision": 1,
      "state": "DELEGATED",
      "claim": "一覧画面のページング実装方式は実装側に委ねる",
      "delegation": {
        "subject": "実装エージェント",
        "operation": "ページング UI/データ取得の実装",
        "scope": "現 plan のみ",
        "expiry": "本 plan 完了時",
        "revocation": "台帳行を UNDECIDED へ戻す"
      }
    },
    {
      "id": "PERF-030",
      "revision": 1,
      "state": "PROVISIONAL",
      "claim": "初期表示は 100 件を上限に仮置きする",
      "reeval_condition": "pilot で実データ件数を観測したら再評価する"
    }
  ]
}
```

## CONTEXT 語彙ファイルテンプレート（JSON・LLM 向け）

`ledger_lint --context` が読む機械可読形式（[context-vocabulary.md](../../shared/references/context-vocabulary.md)）。

```json
{

  "terms": [
    {"id": "T-NAV", "term": "ナビ", "state": "確定"},
    {"id": "T-SESSION", "term": "セッション", "state": "競合中"}
```

## セッション冒頭ブリーフィング テンプレート（人間向け・session）

裁定行の提示に入る前に、何について何件の裁定を求めるか・進め方・途中で中断できることを
3〜5 行の平易な言葉で伝える。台帳やソースを読んでいない人が対象という前提を守り、専門語は使わない:

```
これから {プロジェクト・対象の平易な説明} について {N} 件の裁定をお願いします。
進め方: 前提になるもの → リスクの高いもの の順に、{塊 1 の平易な名前}（{n} 件）→
{塊 2}（{m} 件）のまとまりで聞きます。
所要目安: {数分〜}。疲れたら「保留」と答えればいつでも中断できます（残りは未裁定のまま
保存されます。空回答は承認扱いになる行があるため、中断は必ず「保留」と明示してください）。
```

## 裁定行提示テンプレート（人間向け・session）

1 行 = 1 決定。数十秒で読めるサイズに保つ。同じ領域の行が連続するときは、
その塊の先頭行にだけ領域ラベル行を置く（2 行目以降は省略する）:

```
◆ 画面レイアウトの前提（2 件）

[NAV-001] トップナビはグローバル固定で全画面共通とする
  つまり:   ページ内のどこにいても同じナビを出す、というルールにするかどうかの話
  なぜ今:   画面レイアウトの前提。ここが決まらないと各画面の実装が始められない
  観測事実: 既存モックにヘッダー領域がある
  推奨:     グローバル固定（理由: 画面間の一貫性 / 覆す条件: 画面ごとに異なるナビが要ると判明したら）
  OK すると: 全画面共通のヘッダーコンポーネントとして実装され、画面ごとに違うナビは作らない前提になる
  影響下流: 全画面のレイアウト・ヘッダーコンポーネント

  → OK（承認） / 違う（却下・一言理由） / 任せる（委任） / 保留（未裁定のまま）
```

高リスク行は default を無効化し明示回答を必須にする（Enter で流さない）。

### before/after: 文脈ゼロ・専門語のままの提示をどう直すか

実プロジェクトのパイロットで、次のような行が文脈ゼロ・専門語のまま提示され、
人間が「何の話かわからない」と裁定不能になった。この実例を反面教師にする。

**Before（領域ラベル・つまり・OK すると のいずれも無い — 裁定不能）:**

```
[ENG-006] 同一の排他キーを持つ未終了 run は高々 1 本とする

  → OK（承認） / 違う（却下・一言理由） / 任せる（委任） / 保留（未裁定のまま）
```

**After（領域ラベル + つまり + OK すると で文脈と帰結を供給）:**

```
◆ 定期実行ジョブの多重起動まわり（3 件）

[ENG-006] 同一の排他キーを持つ未終了 run は高々 1 本とする
  つまり:   同じジョブが同時に 2 本走らないようにする、というルールにするかどうかの話
  なぜ今:   重なって走ると副作用（通知送信など）が二重に起きるため
  観測事実: 定期実行と手動実行が同じジョブを同時に起動できる作りになっている
  推奨:     このルールを採用（理由: 二重発行事故の防止 / 覆す条件: 並行実行が必要なジョブが出てきたら）
  OK すると: 前の実行が終わるまで、同じジョブの新しい実行（定時・手動とも）は始まらない
            （承認待ちで止まっている間も次は始まらない点に注意。始まらなかった実行を
            待たせるか取りやめるかは、この主張では決めていないため別途裁定）
  影響下流: スケジューラの起動判定・手動実行ボタンの挙動

  → OK（承認） / 違う（却下・一言理由） / 任せる（委任） / 保留（未裁定のまま）
```

### 生成規約

- 「つまり」「OK すると」を含む提示文は、平易さの合格基準（生成物を読んでいない人が
  一読で「何の話か」を掴める言葉。専門語・コード識別子は開いてから使う）で書く。
  基準の正本は [human-readable-summary.md](../../shared/references/human-readable-summary.md)。
- **「OK すると」は主張が明示的に定めた帰結だけを書く**。主張が決めていない実装挙動
  （例: 開始されなかった実行を待たせるか取りやめるか）を推論で補って断定しない。
  未規定の帰結に触れる必要があるときは「〜は別途裁定」と明示する。読者は正本を
  読まない前提のため、推論で埋めた帰結は検出されないまま誤った承認材料になる。
- 専門語（run・排他キー等）は初出時に一言で開く。CONTEXT 語彙（`term_refs`）に定義がある語は
  その定義を使う。
- 内部トレーサビリティ情報（条項 ID・行 ID 以外の内部参照、「〜として条項化済み」等の記述）は
  人間の判断材料にならないため提示に出さない（詳細送り）。観測事実欄には人間の判断に効く
  観測だけを書く。

## 語彙提示テンプレート（人間向け・語の裁定）

語の意味が競合しているときの 4 択:

```
語「セッション」の意味が競合しています
  候補A: ログインセッション（認証の有効期間）
  候補B: 裁定セッション（この台帳の対話 1 回）
  → 候補A / 候補B / 別語へ分離（両方を別の語にする） / 保留
```

## status 裁定ビュー テンプレート + サンプル出力（人間向け）

冒頭に決定に必要な情報だけを置き、詳細は後送り:

```
未裁定 5 件 / 高リスク 2 件

■ 停止要因（実装を止めている高リスク未裁定）
  - [AUTH-011] 認証の失敗時リダイレクト先が未裁定 → 画面遷移が決められない

■ 高リスク未裁定 上位
  1. [AUTH-011] 認証失敗時リダイレクト先
  2. [DATA-002] 永続データのスキーマ確定

■ 期限切れ委任
  - [IMPL-020] ページング実装（任せた範囲: 現 plan / 期限切れ: 本 plan 完了済み）→ 再確認

■ 次の一手
  裁定セッションを AUTH-011 → DATA-002 の順で回す

■ 再開地点
  前回セッション S-20260721-01 は NAV 系まで完了。AUTH 系が未着手
```

合意済み全件・却下・履歴はこの下に折りたたむ（人間はまず上記だけ読めば動ける）。
