# Review Discussion Flow

チームレビューの議論フローの詳細手順。

## 概要

4段階の議論プロセスを経て、計画の品質を向上させる。

```
Step 1: 独立レビュー（並行） → 各自の専門観点から問題を洗い出す
Step 2: 論点整理（Lead）     → 報告を統合し、議論すべき論点を特定する
Step 3: 議論（全員）         → トレードオフについて多角的に議論する
Step 4: 合意形成（Lead）     → 合意をまとめ、計画ファイルを修正する
```

## Step 1: 独立レビュー（並行）

### 目的

各レビュワーが独立して計画を分析し、自分の専門観点から問題点・懸念を洗い出す。

### 手順

1. 4人のレビュワーを **並行で** Agent spawn する
2. 各レビュワーに以下を渡す:
   - 計画ファイル全文
   - CLAUDE.md の内容
   - `.claude/review-rules.md` の内容（存在すれば）
   - [team-config.md](../../shared/references/team-config.md) に定義されたロール固有のプロンプト
3. 各レビュワーは SendMessage で Lead に報告を送る
4. Lead は全レビュワーの報告を待つ

### Agent プロンプトテンプレート

```
あなたは {role_name} としてチームレビューに参加しています。
チーム名: {team_name}

{role_specific_prompt}

レビューが完了したら、SendMessage ツールを使って Lead（team_name: {team_name}、recipient: "lead"）に結果を報告してください。
```

### タイムアウト

各レビュワーの報告を一定時間待つ。全員の報告が揃わない場合は、届いた報告のみで次に進む（2名以上の報告があれば継続可能）。

## Step 2: 論点整理（Lead）

### 目的

各レビュワーの報告を統合し、議論すべき論点を特定する。

### 手順

1. 全レビュワーの報告を読み込む
2. 以下の3カテゴリに分類する:

| カテゴリ | 説明 | 処理 |
|----------|------|------|
| **共通問題** | 複数レビュワーが独立して指摘した同一の問題 | 高優先度で対応。議論不要なら直接修正 |
| **トレードオフ論点** | レビュワー間で意見が対立する論点 | Step 3 の議論対象 |
| **軽微な問題** | 単独レビュワーのみの指摘で、影響が小さい | 直接修正 or 記録のみ |

3. トレードオフ論点のリストを作成する

### トレードオフ論点の典型例

- **Security vs Performance**: セキュリティチェックがパフォーマンスに影響する
- **Architecture の理想 vs Pragmatist の現実**: 拡張性のための抽象化 vs 工数削減
- **Security vs Usability**: 厳格な検証 vs ユーザー体験
- **Performance vs Maintainability**: 最適化 vs コードの可読性

### SendMessage テンプレート（全員宛）

```
## 論点整理結果

### 共通問題（対応必須）
{common_issues}

### トレードオフ論点（議論対象）
{tradeoff_topics}

### 軽微な問題（記録のみ）
{minor_issues}

以下のトレードオフ論点について、各自の立場から意見を述べてください。
```

## Step 3: 議論（全員）

### 目的

トレードオフが発生する論点について、各レビュワーが自分の立場から意見を述べ、最適解を見つける。

### 手順

1. Lead が論点を全員に共有する（SendMessage）
2. 各レビュワーが自分の立場から意見を述べる（SendMessage で Lead に返信）
3. **各ラウンド終了時にサマリーを出力する**（後述のフォーマット参照）
4. **早期収束条件を確認する**: 全論点が WARN 以下に収束していたら残りのラウンドをスキップ
5. Lead が反論・妥協案を求める（必要に応じて）
6. Lead が議論を収束に導く

### 議論のルール

- 各論点について **最大 3 ラウンド** の往復で収束させる（[共通原則](../../shared/references/team-config.md)参照）
- **早期収束条件**: ラウンド終了時に全論点が WARN 以下に収束していたら残りのラウンドをスキップ（重大度定義は [severity-and-verdicts.md](../../shared/references/severity-and-verdicts.md) 参照）
- 合意に至らない場合は Lead が判断する（理由を明記）
- 全ての BLOCK 指摘は解消するか、解消計画を立てる必要がある
- WARN 指摘は許容可能な理由があれば残してよい

### 議論ラウンドの進め方

```
ラウンド 1:
  Lead → 全員: 論点提示
  全員 → Lead: 各自の意見（SendMessage）
  Lead: ラウンドサマリー出力 + 早期収束チェック

ラウンド 2（必要な場合のみ）:
  Lead → 全員: 反論 or 妥協案の提示
  全員 → Lead: 最終意見（SendMessage）
  Lead: ラウンドサマリー出力 + 早期収束チェック

ラウンド 3（必要な場合のみ）:
  Lead → 全員: 最終確認
  全員 → Lead: 最終意見（SendMessage）
  Lead: ラウンドサマリー出力
```

### ラウンドサマリー出力フォーマット

Lead が各ラウンド終了時にユーザー向けに以下を出力する:

```
── Round {N}/{max} ──
論点: {topic}
  Security: {stance_summary} [BLOCK/WARN/INFO/PASS]
  Performance: {stance_summary} [BLOCK/WARN/INFO/PASS]
  Architect: {stance_summary} [BLOCK/WARN/INFO/PASS]
  Pragmatist: {stance_summary} [BLOCK/WARN/INFO/PASS]
  Status: {resolved | ongoing}
```

## Step 4: 合意形成と計画修正（Lead）

### 目的

議論の結果を踏まえて合意を形成し、計画ファイルを直接修正する。

### 手順

1. 議論結果を整理する
2. 合意事項をリスト化する
3. 計画ファイルを修正する:
   - 問題のあったステップの修正
   - 注意事項の追加
   - エッジケースの追記
   - セキュリティ考慮事項の追加
4. 修正内容のサマリーを生成する

### 判定

判定基準の詳細は [severity-and-verdicts.md](../../shared/references/severity-and-verdicts.md) の「計画レビュー判定」を参照。

| 判定 | 条件 | 次のアクション |
|------|------|---------------|
| APPROVED | 全員が計画の修正版に同意。BLOCK 指摘なし | Phase 2（実装）へ進む |
| APPROVED WITH CONCERNS | 同意だが WARN レベルの残存リスクあり | 残存リスクを記録して Phase 2 へ |
| REJECTED | 根本的な設計変更が必要。BLOCK が解消不可 | cycle 中断、ユーザーに報告 |

### REJECTED 時の処理

1. REJECTED 理由のサマリーを作成する
2. 計画ファイルの Status を `🔴 Rejected` に更新する
3. TeamDelete でチームを解散する
4. ユーザーに報告して cycle を中断する

### 計画修正のフォーマット

計画ファイルに以下のセクションを追加（既存の内容は保持）:

```markdown
## 🔍 Team Review Results

**Reviewed:** {datetime}
**Verdict:** {APPROVED / APPROVED WITH CONCERNS / REJECTED}

### 修正事項
- {修正内容1}（指摘者: {role}）
- {修正内容2}（指摘者: {role}）

### 残存リスク
- {リスク1}（許容理由: {reason}）

### 議論ハイライト
- {論点}: {合意内容の要約}
```

## Step 4.5: ユーザーフィードバック（--interactive 時のみ）

### 目的

`--interactive` フラグ指定時に、ユーザーに議論結果を確認してもらい、コメントを受け付ける。

### フィードバックポイント

Phase 1 合意後（実装前）の **1箇所のみ**。

### 手順

1. レビュー結果を表示し、ユーザーの入力を待つ
2. ユーザー入力を判定:
   - 続行系（「続行」「OK」「問題なし」等）→ Phase 2 へ
   - コメント → Lead が内容を判断して対応

### コメント時の対応

Lead がコメント内容の重大度を判断:

| 性質 | 対応 | チーム利用 |
|------|------|-----------|
| 軽微（スタイル、命名等） | Lead が単独で計画修正 | 不要 |
| 専門的（セキュリティ懸念等） | 該当1名のみに意見を求める | 部分的 |
| 根本的（設計変更） | フルメンバー再議論（最大1回） | 全員 |

### TeamDelete タイミング

`--interactive` 時は、このステップの完了後まで TeamDelete をずらす（チーム再議論の可能性があるため）。

## Step 5: メタレビュー（条件付き）

### 目的

議論で BLOCK が発生した場合に、修正の妥当性とクロスカッティングリスクをチェックする。

### 発火条件

- 議論の合意形成時に **BLOCK が1つでもあった場合のみ** 自動発火
- 全論点が WARN 以下で収束した場合はスキップ

### 手順

メタレビューの条件判定・ラウンド制限・入力制限のルールは [severity-and-verdicts.md](../../shared/references/severity-and-verdicts.md) の「メタレビュー規則」に準拠する。

1. 合意結果の修正差分 + 合意サマリーを作成する
2. **BLOCK/WARN を出したメンバーのみ** に修正確認を SendMessage で依頼する（全員に再送しない）
3. 各メンバーの確認結果を受け取る
4. クロスカッティングリスクを明示的にチェック:
   - 複数提案の組み合わせで発生するリスク
   - 修正が他の合意事項と矛盾していないか
5. Lead が最終判断を下す

### 制約

- **最大1回**: メタレビューで新たな BLOCK が出ても Lead が最終判断を下す
- **入力は差分のみ**: 計画全文の再送は禁止（トークン節約）
- **対象メンバー限定**: BLOCK/WARN を出したメンバーのみ

## エラーケース

### レビュワー spawn 失敗

- 成功したレビュワーが 2 名以上: 残りのメンバーで議論を続行
- 成功したレビュワーが 1 名以下: TeamDelete して cycle 中断

### レビュワーからの報告がない

- 報告がないレビュワーは議論から除外
- 2 名以上の報告があれば続行

### 議論が収束しない

- 最大 3 ラウンドで打ち切り（[共通原則](../../shared/references/team-config.md)参照）
- Lead が最終判断を下す（理由を明記）
