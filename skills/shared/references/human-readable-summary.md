# ヒューマンリーダブル要約 共通契約

レポート生成系スキル（brainstorm wrap / issue create / handoff save / doc-write /
design-guide 等）の**完了報告**が備えるべき「発話サイズの
ヒューマンリーダブル要約」の共通契約。

完了表示に要約ブロックを組み込むスキルは、この契約へ相対 md リンクを張り、下記の
必須要素・体裁・アンチパターン・縮退規定に従う。

## 読者分離の原則（この契約の中心命題）

```
正本（保存されるファイル）  = LLM 向け。密度・網羅性・機械可読性を優先してよい
完了報告（人間が読む接点）  = 人間向け。数分で読める決定密度の高い要約にする
```

保存物の正本は圧縮しない。**人間が読む接点だけ**を平易にする。両者を混同して
正本を薄くしたり、完了報告に正本を逐語再掲したりしてはならない。

認知負荷を下げるのは**行数の圧縮ではなく説明の平易さ**である。短くても
断片・ジャーゴンの羅列では意味がない。行数を足すより言葉を易しくして解く。

## 必須要素

要約ブロックは以下を満たす。

1. **何を作った / やったかの発話サイズ箇条書き** — 生成物の核を 1〜数行で述べる。
2. **人間が確認・裁定すべきポイントの明示** — 承認・判断が要る点、未決定点を挙げる。
   なければ「確認事項: なし」と明示する（省略しない）。
3. **かみ砕いた平易な説明であること** — 専門用語・内部略語・ファイルパスの羅列では
   なく、生成物を読んでいない人が「つまり何なのか」を一読で掴める言葉で書く。
   これが要約の**合格基準**であり、実装者が主観でぶれないよう below の
   before/after ワークト例をアンカーとする。
4. **上限 10 行前後** — 肥大化したら本末転倒。10 行は**上限であって下限ではない**。
5. **かみ砕いた説明が一級要件** — 3 と同旨。圧縮しすぎた断片・体言止めの羅列で
   済ませない。

## 共通体裁と配置（summary-first）

- 要約ブロックは固定ラベル **`📝 つまり:`** で始める。
- 完了表示の**最上部**（ファイルパス・Next Steps より前）に置く。6 スキルで
  視覚的にも配置を統一する（summary-first の配置契約）。

## Before / After ワークト例（主観基準のアンカー）

「一読で掴める」は un-testable な主観基準のため、最低 1 つの before/after を
契約に内蔵し、6 スキルの実装者が同一水準に収束するよう校正する。

**Before（定型のみ — 中身が人間に伝わらない）:**

```
✅ アイデアを保存しました!
📄 File: .agents/artifacts/ideas/20260721_foo.md
📋 Index: .agents/artifacts/ideas/idea-status.md
```

**After（要約付き — 読んでいない人が「つまり何か」を掴める）:**

```
📝 つまり: 完了報告を「✅ + パス」だけにせず、生成物の中身を数行で人間に
   伝える案。承認が儀式化して中身が読まれない問題への対策。
   確認してほしい点: 対象スキルを 6 個に絞ってよいか。

✅ アイデアを保存しました!
📄 File: .agents/artifacts/ideas/20260721_foo.md
📋 Index: .agents/artifacts/ideas/idea-status.md
```

After は保存物を逐語再掲していない。**核と裁定点だけ**を平易な言葉で述べ、
ファイルパスは従来どおり要約ブロックの後に残す。

## 縮退・欠損時の規定

- 要約すべき中身が乏しい時は上限まで埋めず**短く畳む**（定型文水増し禁止の系）。
- 埋められない項目は捏造せず「なし / 未決定」と明示する。
- 要約生成に失敗した場合は、成功用の要約を出さない。
- エコーは逐語転記ではなく**安全な言い換え**である。秘密らしい値（トークン・鍵・
  個人情報）は要約に含めず、省略またはカテゴリ名に置き換える。

## アンチパターン

- 生成物の逐語再掲・網羅列挙
- 定型文の水増し（上限まで無理に埋める）
- 圧縮しすぎた断片・体言止めの羅列
- ジャーゴン・内部略語の無説明使用
- ファイルパスの羅列で中身の説明を代替する
- 秘密情報の新規転記経路を作る

## 「エコー」であり「再生成」ではない

要約は、完了直前に LLM が既に持っている情報（保存した内容そのもの）を発話サイズに
言い換えて表示するだけ。追加の調査・ファイル再読み込みを要求しない（完了報告の
重量化を防ぐ）。

## 既存の良い前例

- **decision-journal**（内容エコー型）: 完了メッセージで宣言した成功基準を 1 行エコー。
- **codebase-review**（summary 先行表示型）: `summary.txt` を完了時に先頭表示。

## plan 系スキルからの被参照（依存逆転のアンカー）

plan 系スキル（plan-create / plan-implement / cycle）の完了表示は、
agreement-ledger plan（`20260721082248`）Phase B2 が**本契約を参照して**実装する。
本契約が要約フォーマットの正本であり、B2 は後からこれに従属する。
