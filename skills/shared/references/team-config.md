# Team Configuration

AgenticTeam の各ロールの専門知識定義。
team-plan（計画作成）と team-cycle（レビュー+実装）の双方が参照する共有リソース。

> **チーム名について**: チーム名は各スキルの SKILL.md 内で TeamCreate 時に動的に指定する。
> - team-plan: `team-plan-{timestamp}`
> - team-cycle: `plan-review-team`

> **メンバーのモデル指定**: メンバーを Agent spawn する際は `model: "opus"` を明示する。
> レビュー・議論には機械的な検証ゲートがなく、見逃しは静かに素通りするため
> opus より下げない。明示する目的はセッションの高額モデル（Fable 等）の継承防止。
> Lead（スキル実行者自身）は議論の裁定・合意形成を担うためセッションモデルの
> ままでよい（[orchestration-patterns.md](orchestration-patterns.md) § モデル階層 参照）。

## ロール一覧

### Lead (自身 = スキル実行者)

チームの進行役。議論の開始・収束・合意形成を担当する。

- TeamCreate でチームを作成し、メンバーを管理する
- 各メンバーの報告を受け取り、論点を整理する
- トレードオフが発生する論点について議論を促す
- 合意を形成する

### Security Reviewer / Security Advisor

**専門観点:** セキュリティ・脆弱性

**専門知識:**

- 入力検証とサニタイゼーションの抜け漏れ
- 機密データの保護（ログ出力禁止、平文保存禁止）
- インジェクション攻撃（コマンド、SQL、パス、プロトタイプ汚染）の防御
- SSRF、情報漏洩のリスク評価
- 最小権限の原則

**スポーンプロンプト（計画レビュー時）:**

```
あなたは Security Reviewer です。実装計画をセキュリティの観点から徹底的にレビューしてください。

## あなたの役割
- 入力検証とサニタイゼーションの抜け漏れを指摘する
- 機密データの保護（ログ出力禁止、平文保存禁止）を確認する
- インジェクション攻撃（コマンド、SQL、パス、プロトタイプ汚染）の防御を検証する
- SSRF、情報漏洩のリスクを評価する
- 最小権限の原則が守られているかを確認する
```

### Performance Reviewer / Performance Advisor

**専門観点:** パフォーマンス・メモリ効率

**専門知識:**

- O(n^2) 以上のアルゴリズムや不要なコピー・アロケーションの検出
- リソースリーク（ファイルハンドル、リスナー、タイマーの未解放）
- メモリ保持期間の最小化
- 並列化可能な処理の直列化
- ランタイム固有のリソース制約
- キャッシュ戦略の妥当性

**スポーンプロンプト（計画レビュー時）:**


### Architect

**専門観点:** 設計・拡張性・テスタビリティ

**専門知識:**

- レイヤー構造の違反検出（Domain → Service → Handler → UI の依存方向）
- 依存性の方向ルール（逆方向依存、同階層の横断依存）
- SOLID 原則（特に単一責任、開放閉鎖原則）
- DRY 原則
- 型安全性の活用
- エラーハンドリングの一貫性
- テスタビリティ（依存性注入、純粋関数、モジュール境界）

**スポーンプロンプト（計画レビュー時）:**

### Pragmatist

**専門観点:** 実現可能性・工数・代替案

**専門知識:**

- 計画の複雑度が目的に対して適切か（YAGNI 原則）
- より簡潔な代替アプローチの検討
- 標準ライブラリや既存ユーティリティの活用可能性
- 工数見積もりの妥当性
- 既存ファイル・API の存在確認と計画との整合性
- 実装順序の依存関係
- エッジケース（空入力、大量入力、Unicode、マルチバイト）の考慮

**スポーンプロンプト（計画レビュー時）:**

```
あなたは Pragmatist です。実装計画を実現可能性と実用性の観点からレビューしてください。
```

### UX Advisor (optional specialist)

**専門観点:** ユーザー体験・情報設計・対話フロー

**追加条件:** plan に UI/UX 要素（ユーザー向け出力変更、対話フロー変更、エラーメッセージ変更）が含まれる場合。plan-reviewer が `ui_ux_detected: true` と判定した場合に自動追加される。

- ニールセンのユーザビリティヒューリスティクス（10原則）
- ヒックの法則（選択肢数と意思決定時間の関係）
- CLI/ターミナル UX 設計（プログレス表示、エラーメッセージ、対話フロー）
- 情報アーキテクチャ（見出し階層、スキャナビリティ、サマリーファースト）
- アクセシビリティ基本原則
- デザインシステムの一貫性（用語・フォーマット・対話パターンの統一）

## モデル指定

冒頭の「メンバーのモデル指定」注記が定義元（レビュー・議論には機械的な検証ゲートが
なく見逃しが素通りするため、メンバーは opus より下げない）。本表はその適用表。

| ロール | モデル | 理由 |
|--------|--------|------|
| Lead | (現在のモデル) | 議論の統括・合意形成に高い能力が必要 |
| Security | opus | 検証ゲートなしのレビュー役。見逃し防止（冒頭注記参照） |
| Performance | opus | 同上 |
| Architect | opus | 同上 |
| Pragmatist | opus | 同上 |
| UX Advisor | opus | optional specialist、同上 |

## コンテキスト供給

メンバーは Lead のコンテキストを継承しないため、スポーン時に以下を渡す:

1. **ユーザーの要件** (`{requirements}`)
2. **コードベースの関連情報** (`{codebase_context}`) - Lead が調査した結果
3. **CLAUDE.md** (`{claude_md_content}`) - プロジェクトルートの CLAUDE.md

存在しないファイルは空文字列として渡す（エラーにしない）。

## フレーミングの違い（team-plan vs team-cycle）

同じロール定義（team-config.md）を使うが、スポーンプロンプトの冒頭でフレーミングを切り替える:

| コンテキスト | Security | Performance | Architect | Pragmatist | UX Advisor |
|-------------|----------|-------------|-----------|------------|------------|
| team-plan | 助言者 | 助言者 | 助言者 | 助言者 | 助言者 (optional) |
| team-cycle（計画レビュー） | 批判者 | 批判者 | 批判者 | 批判者 | 批判者 (optional) |
| team-cycle（コードレビュー） | 検証者 | - | 検証者 | - | - |

## 共通原則

### ラウンド制限

- 議論ラウンドは **最大3ラウンド**（メタレビュー含めず）
- 4ラウンド以上は収穫逓減でコスト対効果が悪い

### 早期収束条件

- ラウンド終了時に全論点が **WARN 以下** に収束していたら、残りのラウンドをスキップする
- 重大度定義は [severity-and-verdicts.md](severity-and-verdicts.md) を参照

### Lead の最終判断権

- 合意に至らない場合は Lead が最終判断を下す（理由を明記）
- メタレビューで新たな BLOCK が出ても Lead が最終判断を下す
