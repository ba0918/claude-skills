# Investigate

完全読み取り専用の軽量調査スキル。問題の原因を特定し、影響範囲を分析し、構造化されたレポートを出力する。
**ファイルの編集は絶対に行わない。** 調査結果に基づく次のアクションはユーザーが判断する。

### 他スキルとの差別化

- **plan / team-plan との違い**: investigate はファイルを一切変更せず、安全に調査だけを行う。計画作成前の事前調査や、実装後の事後検証に最適
- **cycle / team-cycle との違い**: investigate は実装を伴わない。「まず状況を把握したい」「原因が不明」な場面で使う

## 絶対的な制約

### 禁止ツール（いかなる状況でも使用禁止）

- **apply_patch** ツール — ファイル作成・編集禁止

### 許可ツール

- **shell** — **読み取り専用コマンドのみ**（`cat`, `head`, `tail`, `rg`, `grep`, `find` 等。下記の禁止コマンド参照）
- **spawn_agent** — 調査の委譲（読み取り専用エージェント）
- **codex_file_search** — ファイル検索

### shell ツールの制約

以下のコマンドおよびファイルシステム・リポジトリの状態を変更するコマンドは **すべて禁止**:

- `rm`, `rmdir` — ファイル・ディレクトリ削除
- `mv` — ファイル移動・リネーム
- `cp` — ファイルコピー（新規ファイル作成になりうる）
- `chmod`, `chown` — 権限変更
- `git commit`, `git push`, `git reset`, `git checkout --` — リポジトリ状態変更
- `tee`, リダイレクト (`>`, `>>`) — ファイル書き込み
- `sed -i`, `awk -i inplace` — インプレース編集
- `touch`, `mkdir` — ファイル・ディレクトリ作成

許可される shell コマンドの例: `git log`, `git diff`, `git show`, `git blame`, `git status`, `ls`, `cat`, `head`, `tail`, `wc`, `diff`, `file`, `stat`, テストの実行 (`npm test`, `pytest` 等)

### セキュリティ上の注意

調査対象のコードに機密情報（API キー、パスワード、トークン等）が含まれる場合、レポートにそのまま出力しないこと。存在を報告するにとどめ、値は伏せる。

## フロー

```
問題の説明 → コンテキスト取得 → 多角的調査 → レポート出力
```

## Phase 1: コンテキスト取得

1. ユーザーの問題説明を `$ARGUMENTS` から取得
2. プロジェクトの構造を把握（CLAUDE.md / AGENTS.md、ディレクトリ構成）
3. 関連するエラーメッセージ、ログ、スタックトレースがあれば特定

## Phase 2: 多角的調査

`spawn_agent` で読み取り専用の調査エージェントを起動し、以下を調査する:

1. **問題の再現確認** — エラーが再現可能か、条件は何か
2. **原因の特定** — コードを追跡し根本原因を突き止める
3. **影響範囲の分析** — 同じパターンが他にも存在するか、依存関係は何か
4. **関連するテスト** — 既存テストの有無、テストが問題をカバーしているか

調査が広範囲にわたる場合は、`spawn_agent` で並行調査を行う。

### Agent エラー時のフォールバック

`spawn_agent` がエラーで失敗した場合は、再試行せずに shell（読み取り専用コマンド）での直接調査に切り替える。

## Phase 3: 調査レポート出力

以下の構造で **会話上に** レポートを出力する（ファイルには書き出さない）:

```
══════════════════════════════════════
INVESTIGATION REPORT
══════════════════════════════════════

## 1. 問題の概要

{何が起きているかの簡潔な説明}

## 2. 原因分析

{なぜ起きているかの根本原因}
- 直接原因: {直接的なコード上の原因}
- 根本原因: {設計レベルの原因（該当する場合）}

## 3. 影響範囲

- 影響ファイル: {ファイルリスト}
- 影響機能: {機能リスト}
- 同様のパターン: {他に同じ問題が存在するか}

## 4. 確信度

{高 / 中 / 低} — {判定理由}

## 5. 修正案

{どう直すべきかの具体的な説明。コード変更の方針を示すが、実行はしない}

### 案 A: {アプローチ名}
- 変更箇所: {ファイル・行}
- 概要: {何をどう変えるか}
- メリット/デメリット: {トレードオフ}

### 案 B: {アプローチ名}（代替案がある場合）
- ...

## 6. 推奨アクション

{以下の表から状況に最も適したアクションを選んで提案}
```

### 推奨アクションの分類

| 状況 | 推奨アクション | コマンド例 |
|------|---------------|-----------|
| 軽微な修正（1-2ファイル、明確な修正） | そのまま修正 or iterate | `$iterate {修正内容の要約}` |
| スコープ外・後で対応 | issue として記録 | `$issue create {問題のタイトル}` |
| 中〜大規模な修正 | 計画を立ててサイクル | `$plan` → `$cycle` |
| 実装後の追加修正 | iterate で軽量改善 | `$iterate {追加修正の指示}` |
| 判断材料不足・要議論 | 追加調査や議論を継続 | （会話を続ける） |

**重要:** 推奨アクション出力時は、上記コマンド例をそのままコピペして実行できる形式で提示すること。ユーザーが次のアクションに即座に移れるようにする。

## 事後検証モード

実装・コミット完了後に「期待通りに実装されたか」「意図しない副作用がないか」を検証する場合にも使用できる。

### トリガー

「検証して」「動作確認して」「実装確認」「本当にこれで正しいか確認」

### フロー

1. 実装結果のコードを調査する（変更されたファイルの内容を確認）
2. 期待される動作との差分を分析
3. 副作用や見落としがないかを確認
4. 差分があれば影響範囲を特定し、追加修正の推奨アクションを提示

## 重要なルール

- **調査結果を見つけても絶対に修正を始めない** — これがこのスキルの存在意義
- **推奨アクションは提案のみ** — 勝手に実行しない
- **不確かなことは不確かと報告する** — 確信度を正直に表明する
- **過度に広い調査は避ける** — ユーザーの問題に集中し、スコープを絞る
