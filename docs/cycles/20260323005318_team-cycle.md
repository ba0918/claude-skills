# Team Cycle スキル

**Cycle ID:** `20260323005318`
**Started:** 2026-03-23 00:53:18
**Status:** 🔵 Implementing

## 📝 What & Why

既存の cycle ワークフローのレビューフェーズを AgenticTeam 機能で強化する新スキル `team-cycle` を作成する。

現在の cycle では plan-reviewer が1エージェントで6観点を順番にレビューしているが、1エージェントが「書く側」と「批判する側」を兼任するためバイアスが生じやすい。AgenticTeam を使い、異なる専門観点を持つ複数レビュワーが**議論しながら**レビューすることで、トレードオフの検討が自然に発生し、計画の品質が向上する。

実装フェーズは既存の plan-implement をそのまま使い、変更範囲を最小限に抑える。

## 🎯 Goals

- AgenticTeam によるチーム議論型レビューフェーズの実現
- 既存の cycle / plan-implement との互換性維持
- 既存のファイル形式（docs/cycles/、docs/status.md、結果ファイル）をそのまま使用

## 📐 Design

### アーキテクチャ概要

```
team-cycle コマンド
  │
  ├─ Phase 0: 準備（計画ファイル特定・検証）
  │
  ├─ Phase 1: チームレビュー（AgenticTeam）★新規
  │    │
  │    ├─ TeamCreate: レビューチーム作成
  │    │
  │    ├─ Agent spawn × 4（チームメンバー）:
  │    │    ├─ Security Reviewer
  │    │    ├─ Performance Reviewer
  │    │    ├─ Architect
  │    │    └─ Pragmatist
  │    │
  │    ├─ タスク管理:
  │    │    ├─ Task 1: 各自独立レビュー → 結果を Lead に報告
  │    │    ├─ Task 2: Lead が論点を整理し議論を開始
  │    │    ├─ Task 3: メンバー間で議論（トレードオフ検討）
  │    │    └─ Task 4: Lead が合意をまとめ計画を修正
  │    │
  │    ├─ 議論結果で計画ファイルを直接修正
  │    └─ TeamDelete: チーム解散
  │
  ├─ Phase 2: 実装（既存の plan-implement を Agent 委譲）
  │
  └─ Phase 3: 完了処理
       ├─ 結果ファイル生成（docs/cycles/results/）
       ├─ status.md 更新（Current Session クリア）
       ├─ session-history.md アーカイブ
       ├─ commit（claude-skills:commit）
       └─ issue close（計画に Issue 参照があれば）
```

### チーム構成

| ロール | 専門観点 | レビュー担当 | キャラクター |
|--------|----------|-------------|-------------|
| **Lead** (自身) | 全体統括 | 議論の進行・合意形成・計画修正 | 中立的な進行役 |
| **Security Reviewer** | セキュリティ・脆弱性 | 入力検証、機密データ保護、インジェクション対策 | 慎重派、リスクに厳しい |
| **Performance Reviewer** | パフォーマンス・メモリ | O(n²)検出、メモリリーク、キャッシュ戦略 | 効率重視、計測で語る |
| **Architect** | 設計・拡張性・テスタビリティ | レイヤー違反、依存性方向、SOLID原則 | 長期視点、原則重視 |
| **Pragmatist** | 実現可能性・工数・代替案 | 複雑すぎないか、より簡潔な方法はないか | 現実主義、YAGNI派 |

> 既存 plan-reviewer の6観点（Feasibility / Security / Performance / Architecture / Completeness / Alternatives）を4ロールに再編。Feasibility と Alternatives は Pragmatist に、Completeness は全員の横断的責任とする。

### レビュー議論フロー

```
Step 1: 独立レビュー（並行）
  各レビュワーが計画を読み、自分の専門観点から問題点・懸念を洗い出す
  → 結果を Lead に SendMessage で報告

Step 2: 論点整理（Lead）
  Lead が各レビュワーの報告を整理し、以下を特定:
  - 複数レビュワーが指摘した共通問題
  - レビュワー間でトレードオフが発生する論点
  - 単独で解決可能な軽微な問題
  → 議論すべき論点をチーム全体に共有

Step 3: 議論（全員）
  トレードオフが発生する論点について議論
  - Security vs Performance のトレードオフ
  - Architecture の理想 vs Pragmatist の現実
  - 各自が自分の立場から意見を述べ、反論・妥協案を出す
  → Lead が議論を収束に導く

Step 4: 合意形成と計画修正（Lead）
  - 合意事項をまとめる
  - 計画ファイルを直接修正（問題のあったタスクの修正、注意事項の追加等）
  - 修正内容のサマリーを生成
```

### 判定ロジック

チームレビューでは既存のスコアリング（0-100）ではなく、**合意ベースの判定**を使用：

| 判定 | 条件 | 次のアクション |
|------|------|---------------|
| ✅ APPROVED | 全員が計画の修正版に同意 | Phase 2（実装）へ進む |
| ⚠️ APPROVED WITH CONCERNS | 同意だが残存リスクあり | 残存リスクを記録して Phase 2 へ |
| 🛑 REJECTED | 根本的な設計変更が必要 | cycle 中断、ユーザーに報告 |

### エラーハンドリング（Phase 1）

Phase 1 の全処理は **TeamDelete を保証する try-finally パターン** で実装する:

1. TeamCreate 成功後、以降のどの段階でエラーが発生しても TeamDelete を必ず実行する
2. Agent spawn が部分的に失敗した場合:
   - 成功したエージェントのみで議論を続行する（2名以上いれば継続可能）
   - 1名以下の場合は TeamDelete 後に cycle 中断
3. REJECTED 判定時の後処理:
   - TeamDelete でチーム解散
   - 計画ファイルの Status を `🔴 Rejected` に更新
   - REJECTED 理由のサマリーをユーザーに表示
   - Phase 2 には進まず cycle 中断

### Key Points

- **既存 cycle との共存**: `team-cycle` は完全に新規スキルとして追加。既存の `cycle` は一切変更しない
- **チームメンバーのモデル指定**: トークンコスト抑制のため、レビュワーには `sonnet` を使用（Lead のみ現在のモデル）
- **コンテキスト供給**: メンバーは Lead のコンテキストを継承しないため、スポーン時に計画ファイル全文 + プロジェクトの CLAUDE.md + review-rules.md（存在すれば）を渡す
- **実装フェーズの再利用**: Phase 2 は既存の `claude-skills:plan-implement` をそのまま Agent 委譲で呼び出す
- **結果ファイル互換**: `docs/cycles/results/` に生成する結果ファイルは既存フォーマットを踏襲。Refine セクションにチーム議論のサマリーを追加
- **環境変数チェック**: Phase 1 開始前に `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` の設定を確認し、未設定なら案内を出して中断
- **Phase 3 完了処理の詳細**: 既存 cycle.md の Phase 3 と同等の手順を踏む:
  1. 結果ファイル生成: `docs/cycles/results/{plan_basename}_result.md`（ディレクトリがなければ `mkdir -p`）
  2. status.md 更新: `skills/plan/references/status-update-guide.md` の Case 2 に従い、session-history.md へアーカイブ → Session History セクションクリア → Current Session クリア
  3. コミット: `claude-skills:commit` で成果物をコミット
  4. Issue close: 計画ファイルに `**Issue:**` 行があれば `claude-skills:issue` の `close {slug}` を実行（失敗しても cycle 自体は成功扱い）
- **TeamDelete の保証**: TeamCreate 以降の全処理を try-finally パターンで保護し、エラー・REJECTED・正常完了のいずれでも TeamDelete を必ず実行する
