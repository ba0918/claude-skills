## What & Why

Claude Code のセッションデータから「スキル使用時の摩擦」を構造的シグナルとして検出・分析し、AgenticTeam で改善方法を議論した上でスキル改善を実行するメタスキルを構築する。スキルが20個以上に成長した今、手動での品質追跡は限界に達しており、データ駆動の自己改善ループが必要。

## Goals

- セッションデータから摩擦シグナル（リトライ、修正指示、セッション離脱等）を自動検出する collect スクリプトを実装する
- 4エージェント並行による摩擦分析パイプラインを構築する（codebase-review パターン流用）
- Dry-run + 人間レビューゲート付きの安全な改善実行フローを実現する
- 既存スキルパターン（investigate, iterate, team-cycle）を最大限再利用し、新規実装を最小化する

## Design

### Architecture Overview

```
Phase 1: データ収集（scripts/collect.py → Bash実行 → JSON出力）
  │
  ↓ .claude/tmp/skill-improve-{datetime}/context.json
  │
Phase 2: 摩擦分析（4エージェント並行 — codebase-review パターン）
  │  friction-detector / pattern-analyzer / expectation-auditor / drift-detector
  ↓ .claude/tmp/skill-improve-{datetime}/friction-report.md
  │
Phase 3: 改善仮説生成（investigate パターン — 読み取り専用）
  │  friction-report.md → 改善仮説 A/B/C 生成
  ↓ iterate サイズ判定で自走度決定
  │
Phase 4: 改善実装（iterate or team-cycle に委譲）
  小規模 → iterate（SKILL.md 直接修正）
  大規模 → team-cycle（friction-report をコンテキスト注入して議論）
```

### Files to Create/Change

```
skills/skill-improve/
  SKILL.md                    - メインスキル定義（4フェーズのオーケストレーション）
  references/
    friction-schema.md        - 摩擦分析の JSON スキーマ定義
    analysis-roles.md         - 4分析エージェントのロール定義
    scoring-guide.md          - 摩擦スコアリング基準
  scripts/
    collect.py                - セッションデータ収集スクリプト（Python）

commands/

```

### Key Points

- **collect.py の設計**: tokoroten/prompt-review の collect.py を参考に、skill-improve 固有のフィールド（skill_invocations, friction_signals）を追加。メッセージ本文は収集しない（構造的シグナルのみ）
- **セキュリティ**: シークレット検出+マスク（JONLパース直後に実行）、シンボリックリンク対策（許可ルート: `~/.claude/projects/`）、パス正規化、デフォルトスコープ制限（現プロジェクト+30日）。`--all` フラグは廃止（スコープ制限の設計意図と矛盾するため）
- **抽象化レイヤー**: 初期実装では3層抽象化（SessionParser→SessionEvent→FrictionAnalyzer）は行わず、collect.py 内にフラットに実装する。将来の拡張時に抽象化を導入する。JSONL読めない場合は session-history.md + git log の軽量モードにフォールバック（ただし軽量モードでは retry_count と turns_to_completion のみ推定可能であることを明記する）
- **自走度制御**: iterate のサイズ判定を転用。摩擦スコア低→レポートのみ / 中→iterate Small / 高→team-cycle。全レベルで Dry-run 必須

## Implementation Steps

### Step 1: collect.py の実装
**Files:** `skills/skill-improve/scripts/collect.py`

- ~/.claude/projects/ 配下の JSONL セッションデータを読み取り
- スキル呼び出しシグナル（`/claude-skills:` プレフィックス、Skill ツール呼び出し）を検出
- 摩擦シグナルを構造的に抽出（**メッセージ本文に依存しないシグナルのみ**）:
  - `retry_count`: 同一スキルの連続呼び出し回数
  - `correction_turns`: スキル実行後の修正指示ターン数（ツール呼び出しパターンから推定）
  - `session_abandoned`: セッション完了前の離脱
  - `tool_error_count`: ツール実行エラーの回数（本文不要・構造シグナル）
  - `turns_to_completion`: スキル完了までのターン数
- シークレット検出+マスク処理（**JONLパース直後・フィールド抽出前に実行**。順序: シークレット検出 → マスク → テキスト切り捨て）
- シンボリックリンク検証（realpath + prefix check、**許可ルート: `~/.claude/projects/`**）
- JSONL 読み取りは**ストリーミング**（1行ずつ処理、全行メモリ保持しない）
- 引数: `--days N`（デフォルト30）, `--project NAME`（デフォルト: cwd から推定）
- ~~`--all`（全プロジェクト横断）~~ → 廃止。明示的な `--project NAME` 指定のみ
- 出力: JSON（summary + skill_invocations + friction_signals + secret_warnings）

### Step 2: SKILL.md の作成（オーケストレーション層）
**Files:** `skills/skill-improve/SKILL.md`, `commands/skill-improve.md`

- Workflow Selection: `analyze`（デフォルト）/ `report`（レポートのみ）/ `improve`（改善実行付き）
- Phase 1: Bash で collect.py 実行 → context.json に書き出し
- Phase 2: 4エージェント並行起動（Agent ツール）
  - friction-detector: リトライ・修正パターンの抽出
  - pattern-analyzer: iterate 多重起動・同一エラー繰り返しの頻度分析
  - expectation-auditor: スキル定義の期待値とユーザー実際の使い方のギャップ分析
  - drift-detector: スキル設計当初の想定ユースケースからのドリフト検出
- Phase 3: 統合エージェントが friction-report.md を生成（investigate パターン）
- Phase 4: `improve` モード時のみ、iterate or team-cycle に委譲

### Step 3: references/ の作成
**Files:** `skills/skill-improve/references/friction-schema.md`, `analysis-roles.md`, `scoring-guide.md`

- friction-schema.md: collect.py の出力 JSON スキーマ + friction-report.md のスキーマ
- analysis-roles.md: 4分析エージェントのスポーンプロンプト（codebase-review の roles 定義と同型）
- scoring-guide.md: 摩擦スコアリング基準（閾値テーブル + アクション対応表）

### Step 4: CLAUDE.md とドキュメント更新
**Files:** `CLAUDE.md`

- 主要スキル一覧に skill-improve を追加
- コマンド→スキル対応表に追加
- ワークフロー設計パターンに「スクリプト経由データ収集」パターンを追記

### Step 5: 動作確認 + アイデアステータス更新
**Files:** `docs/ideas/idea-status.md`, `docs/ideas/2026-03-23_skill-improve.md`

- analyze モードでフル分析パイプラインを実行
- アイデアの Status を Idea → Planned に更新、archives/ に移動
