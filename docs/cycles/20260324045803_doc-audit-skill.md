## What & Why

プロジェクト内の docs（plan, issue, idea 等）が放置・腐敗するのを検出し、自動修復またはユーザー確認で解決する「doc-audit」スキルを新規作成する。現状、各スキルが個別にアーカイブ処理を持つが横断的なクリーンアップ手段がなく、ステータスの浮遊・重複 issue の放置・plan の進捗未更新が発生している。

## Goals

- docs 内の全アーティファクト（ideas, issues, cycles, session-history）を横断的にスキャンし、不整合を検出する
- 機械的に修復可能な問題（ステータス更新、アーカイブ移動、履歴追記）は自動修復する
- 意味的判断が必要な問題（放置 plan の判定、重複 issue の統合）はユーザーに確認する
- --dry-run モードでレポートのみの実行をサポートする

### Key Points

- **独立スキルとして新規作成**: doc-check はコード↔ドキュメントの整合性、doc-audit はドキュメント↔ドキュメントの整合性。対象が異なるため責務を分離する
- **単一 Agent で逐次チェック**: 現状のアーティファクト量（cycles 18, ideas 2, issues 0）では並行化不要。YAGNI 原則を優先する
- **検出ルールのモジュール化**: references/checks.md に各ルールを独立定義。将来の検出ルール追加は OCP に従い新ルールの追加のみで対応
- **AUTO_FIX / NEEDS_JUDGMENT の明確な基準**:
  - AUTO_FIX: 機械的に検証可能 + 冪等 + データ損失リスクなし（ステータス更新、archives/ 移動、session-history 追記）
  - NEEDS_JUDGMENT: 意味的判断が必要（放置 plan の判定、重複 issue の統合）
- **--dry-run モード**: Fix フェーズをスキップし Report のみ実行。安全な初回実行を保証

### Architecture

```
doc-audit コマンド ($ARGUMENTS: "--dry-run" | "" | カテゴリ指定)
  │
  ├─ Phase 1: Scan
  │   ├─ docs/ideas/ スキャン（idea-status.md + 個別ファイル + archives/）
  │   ├─ docs/issues/ スキャン（issue-status.md + 個別ファイル + archives/）
  │   ├─ docs/cycles/ スキャン（plan ファイル + results/）
  │   ├─ docs/status.md スキャン
  │   └─ docs/session-history.md スキャン
  │   → 全アーティファクトのメタデータ収集
  │
  ├─ Phase 2: Analyze
  │   ├─ references/checks.md の各ルールを順次適用
  │   ├─ 各問題を {type, severity, auto_fix, description, affected_files, fix_action} で記録
  │   └─ 問題なしの場合は「健全」と判定
  │   → 問題リスト生成
  │
  ├─ Phase 3: Report
  │   ├─ references/report-template.md に従いレポート出力
  │   ├─ AUTO_FIX / NEEDS_JUDGMENT の分類表示
  │   ├─ 問題件数のサマリー
  │   └─ --dry-run の場合はここで終了
  │
  └─ Phase 4: Fix
      ├─ AUTO_FIX 問題を自動修復（ステータス更新、archive 移動、履歴追記）
      ├─ NEEDS_JUDGMENT 問題を1件ずつユーザーに直接質問して確認
      │   ├─ ユーザーが承認 → 修復実行
      │   └─ ユーザーがスキップ → 次の問題へ
      └─ 修復結果のサマリー出力
```

### Detection Rules（references/checks.md に定義）

| # | ルール名 | 検出対象 | 判定 |
|---|---------|---------|------|
| 1 | stale-idea-status | idea-status.md のステータスが実態（cycle 存在・スキル実装）と不一致 | AUTO_FIX |
| 2 | unarchived-idea | Planned/Implemented な idea が archives/ に移動されていない | AUTO_FIX |
| 3 | orphan-plan-in-status | status.md に残っている plan が session-history.md に未記載 | AUTO_FIX |
| 4 | missing-session-history | 完了済み cycle が session-history.md に記載されていない | AUTO_FIX |
| 5 | abandoned-plan | Planning/In Progress のまま放置された plan（新しい plan が開始済み） | NEEDS_JUDGMENT |
| 6 | duplicate-issue | 意味的に重複する Open issue | NEEDS_JUDGMENT |
| 7 | stale-issue-status | issue-status.md に記載されているが対応 cycle で解決済みの issue | NEEDS_JUDGMENT |
| 8 | index-file-mismatch | インデックスファイル（idea-status.md, issue-status.md）と実ファイルの不一致 | AUTO_FIX |
| 9 | result-accumulation | docs/cycles/results/ の件数警告（v1 はレポートのみ、修復なし） | REPORT_ONLY |
| 10 | date-format-legacy | 旧日付フォーマット（YYYY-MM-DD）のファイル名検出 | NEEDS_JUDGMENT |

### Scan Phase の詳細

各ディレクトリのスキャンで収集する情報:

**ideas/**:
- idea-status.md のテーブルエントリ一覧
- 個別 idea ファイルの存在とメタデータ（タイトル、ステータス）
- archives/ 内のファイル一覧
- 対応する cycle の有無（ファイル名やタイトルでマッチング）
- 対応するスキルの実装有無（skills/ ディレクトリ検索）

**issues/**:
- issue-status.md のテーブルエントリ一覧
- 個別 issue ファイルの存在
- archives/ 内のファイル一覧
- 対応する cycle の有無

**cycles/**:
- 全 plan ファイル一覧（タイムスタンプ、slug、Status 行）
- results/ 内のファイル一覧と件数
- session-history.md に記載されているかの照合

**status.md**:
- Current Session の有無
- アクティブセッションの plan ファイル存在確認

### Error Handling

各フェーズで想定されるエラーケースと対処:

| ケース | 対処 |
|--------|------|
| `docs/` ディレクトリが存在しない | スキャン開始前にチェックし、「対象ディレクトリが見つかりません」と報告して終了 |
| `idea-status.md` / `issue-status.md` が存在しない | 該当カテゴリのスキャンをスキップし、レポートに「インデックスファイル不在」を記録 |
| `docs/status.md` / `docs/session-history.md` が存在しない | 該当チェックをスキップし、レポートに記録 |
| Markdown テーブルのパース失敗（フォーマット不正） | 該当ファイルのルールチェックをスキップし、WARN として「パース不可」を報告 |
| AUTO_FIX 時のファイル書き込みエラー | エラーを記録し、次の問題に進む（部分的修復の許容） |
| archives/ ディレクトリが存在しない | 移動先ディレクトリを作成してから修復を実行 |

### 合意事項
- 独立スキルとして新規作成（doc-check とは責務が異なる）
- 単一 Agent で逐次チェック（並行化は YAGNI）
- Scan → Analyze → Report → Fix の4段階フロー
- 検出ルールのモジュール化（references/checks.md に独立定義）
- AUTO_FIX / NEEDS_JUDGMENT の明確な基準（機械的・冪等 = AUTO_FIX、意味的判断 = NEEDS_JUDGMENT）
- --dry-run モード対応
- Cycle result は v1 スコープ外（レポートで件数警告のみ）

### 議論ハイライト
- **独立 vs 拡張**: 全員が独立スキルを支持。doc-check はコード↔ドキュメント、doc-audit はドキュメント↔ドキュメントで対象が異なる
- **並行化**: Performance + Architect + Pragmatist が不要と判断。現状のアーティファクト量（20-50個）では単一 Agent で十分
- **AUTO_FIX の境界**: Security の慎重アプローチと Pragmatist の積極アプローチの間で合意。ファイル移動は git 復元可能なため AUTO_FIX に含める。放置 plan と重複 issue は意味的判断が必要なため NEEDS_JUDGMENT
- **Cycle result 保持**: 全員が v1 スコープ外で合意。YAGNI 原則を優先

### 各メンバーの貢献
- **Security Advisor**: AUTO_FIX の境界線を慎重に定義。冪等性とデータ損失リスクの判断基準を提供。パストラバーサル防止の指摘
- **Performance Advisor**: 並行化不要の根拠を提示（アーティファクト量 vs オーバーヘッド）。result 蓄積問題の将来リスクを指摘
- **Architect**: 検出ルールのモジュール化（OCP 準拠）を提案。レイヤー設計として Scan/Analyze/Report/Fix の分離を推奨。冪等性を AUTO_FIX の基準に
- **Pragmatist**: YAGNI に基づくスコープ制御。既存パターンとの一貫性を確保。v1 で必要十分な範囲の明確化
