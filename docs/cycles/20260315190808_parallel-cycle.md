# Parallel Cycle (指示分解 + 並行サイクル実行)

**Started:** 2026-03-15 19:08:08
**Status:** 🟡 In Progress

## 📝 What & Why

雑な自然言語の指示を複数の plan に自動分解し、独立した cycle を worktree 上の別ブランチで並行実行、完了後に自動マージするオーケストレータを追加する。
現状は「1指示 = 1 plan = 1 cycle」の直列フロー。複合的な機能要求を1コマンドで分解→並行実行できれば、大幅な時間短縮と認知負荷の軽減が見込める。

## 🎯 Goals

- 自然言語の指示を複数の独立した plan に自動分解できる
- 分解時に依存関係と優先順位を自動判定する
- 独立した cycle を並行実行できる
- コンフリクトが原理的に発生しない安全な設計
- 既存の cycle/plan インフラをそのまま再利用する
- 失敗した cycle が他の cycle に影響しない

## 🚧 破綻しないための設計原則

### 原則 1: ファイル直交性の事前保証

**コンフリクトを解決するのではなく、コンフリクトが発生しない状態を保証する。**

```
Plan A の影響ファイル: {a1.ts, a2.ts, a3.test.ts}
Plan B の影響ファイル: {b1.ts, b2.ts, b3.test.ts}
Plan C の影響ファイル: {c1.ts, a1.ts}  ← A と交差！

判定:
  A ∩ B = ∅  → 並行 OK
  A ∩ C = {a1.ts} → 並行 NG
  B ∩ C = ∅  → 並行 OK

実行グループ:
  Group 1: [A, B] を並行実行
  Group 2: [C] を Group 1 完了後に実行
```

各 plan の「Files to Change」セクションからファイル一覧を抽出し、集合の交差判定を行う。
交差がある cycle 同士は同じグループに入れず、直列化する。

### 原則 2: Worktree 隔離

各 cycle は EnterWorktree ツールで専用の worktree を作成し、その中で Agent ツールを起動して実行する。
- EnterWorktree で worktree を作成 → Agent で cycle 実行 → ExitWorktree で worktree を削除
- 物理的にブランチが分離される
- 1つの cycle が壊れても他に影響しない
- EnterWorktree の戻り値からブランチ名と worktree パスが取得できる

> **注意**: Agent ツール自体には `isolation` パラメータがない。worktree の作成・削除は EnterWorktree / ExitWorktree ツールでオーケストレータ側が明示的に管理する。

### 原則 3: Fast-Forward マージのみ

ファイル直交性が保証されているため、マージは必ず成功する。
ただし安全策として：
- `git merge --no-ff` で各ブランチをマージ
- マージ後に自動テストを実行
- テスト失敗時はそのブランチのマージを revert し、成功分だけ残す

### 原則 4: 部分成功の許容

3つの cycle のうち1つが失敗しても、残り2つの成果は保持する。

```
Cycle A: ✅ Success → マージ済
Cycle B: ❌ Refine BLOCK → マージしない、ブランチは残す
Cycle C: ✅ Success → マージ済

結果: A と C の成果が main に反映。B は手動対応待ち。
```

## 📐 Design

### 全体フロー

```
parallel-cycle コマンド
  │
  ├── Phase 0: 指示の分解（Decompose）
  │     自然言語の指示を受け取る
  │     → 機能単位に分解
  │     → 依存グラフを構築
  │     → 各単位ごとに plan ファイルを生成（既存 /plan-create 相当）
  │     → ユーザに分解結果を提示して承認を得る
  │
  ├── Phase 1: 直交性チェック & グループ化
  │     各 plan の影響ファイルを抽出 → 交差判定
  │     依存グラフ + 直交性 → 実行グループを決定
  │
  ├── Phase 2: 並行実行（グループごと）
  │     Group N の各 cycle に対して EnterWorktree → Agent で cycle 実行
  │     並行実行数は最大 3 に制限（API レート制限への対処）
  │     各 Agent は既存の cycle スキルをそのまま実行
  │
  ├── Phase 3: マージ
  │     成功した cycle のブランチを順次マージ
  │     マージ後にテスト実行 → 失敗なら revert
  │
  └── Phase 4: サマリー
        全 cycle の結果を統合レポート
```

### Phase 0: 指示分解の詳細設計

これが最も重要なフェーズ。LLM の強みである自然言語理解を活かす。

#### 分解の入力と出力

```
入力: 「ユーザ認証して、体重/体脂肪を入力し、グラフに反映。AIと対話で行動記録」

出力:
  Plan A: ユーザ認証 (signup/login/session)
    影響ファイル: auth/*, middleware/*
    依存: なし（ルート）
    優先度: 1（他が依存するため最優先）

  Plan B: 体重・体脂肪 CRUD
    影響ファイル: models/weight.*, routes/weight.*, components/weight-form.*
    依存: [A]（認証済ユーザが前提）
    優先度: 2

  Plan C: グラフ表示
    影響ファイル: components/chart.*, utils/chart-data.*
    依存: [B]（データが必要）
    優先度: 3

  Plan D: AI対話 + 行動記録
    影響ファイル: services/ai.*, models/activity.*, components/chat.*
    依存: [A]（認証済ユーザが前提）
    優先度: 2

依存グラフ:
  A → B → C
  A → D

実行グループ:
  Group 1: [A]
  Group 2: [B, D]  ← 並行可能（ファイル直交）
  Group 3: [C]
```

#### 分解の原則

1. **機能境界で切る**: 1つの plan = 1つのユーザ向け機能
2. **データフローで依存を判定**: 「B は A が作るテーブル/API を使う」→ B depends on A
3. **影響ファイルは保守的に推定**: 迷ったら広めに取る（並行不可になるだけで壊れはしない）
4. **共有インフラ（DB マイグレーション等）は最初の plan に含める**

#### エッジケースのフォールバック

- **0 plan に分解**: 指示が不明瞭で分解不能な場合、エラーメッセージを表示し、指示の明確化を求めて終了する。
- **1 plan に分解**: 並行実行の必要がないため、通常の `/cycle` にフォールバックする（parallel-cycle のオーバーヘッドを回避）。
- **影響ファイルが空の plan**: 直交性チェックでは「他の全 plan と直交」として扱い、任意のグループに配置可能とする。

#### ユーザ承認ステップ

分解結果をユーザに提示し、承認を得てから実行に進む。

```
══════════════════════════════════════
DECOMPOSE RESULT
══════════════════════════════════════

Plans: 4
Execution groups: 3

Group 1 (sequential):
  [A] ユーザ認証 — 依存なし

Group 2 (parallel):
  [B] 体重・体脂肪 CRUD — depends on A
  [D] AI対話 + 行動記録 — depends on A

Group 3 (sequential):
  [C] グラフ表示 — depends on B

Estimated total groups: 3 rounds
──────────────────────────────────────
Proceed? (y/n/edit)
```

`edit` を選んだ場合: 分解の修正指示を受け付けて再分解。
**これが唯一のユーザ確認ポイント。** 以降はヘッドレスで全自動。

### Key Points

- **既存の cycle をそのまま再利用**: parallel-cycle は cycle を呼び出すだけ。cycle 自体は変更しない
- **既存の plan テンプレートをそのまま再利用**: 分解後の各 plan は `/plan-create` と同じフォーマットで生成
- **EnterWorktree / ExitWorktree**: Claude Code の既存ツールで worktree のライフサイクルを管理
- **plan テンプレートへの依存**: 各 plan の「Files to Change」セクションが直交性チェックの入力になる
- **グループ化による段階実行**: 依存グラフ + ファイル直交性の両方を考慮してグループを決定
- **ユーザ承認は1回だけ**: Phase 0 の分解結果確認のみ。以降は全自動

### コマンドインターフェース

```bash
# 自然言語から分解 → 並行実行（メインの使い方）
/parallel-cycle ユーザ認証して、体重を記録して、グラフに出して、AIと対話

# 既存の plan ファイルを直接指定して並行実行（分解スキップ）
/parallel-cycle docs/cycles/xxx.md docs/cycles/yyy.md docs/cycles/zzz.md
```

引数が `.md` パスならスキップして Phase 1 から開始。それ以外は自然言語とみなして Phase 0 から。

### オーケストレータの責務（薄く保つ）

1. **分解** — 自然言語 → 複数 plan 生成（Agent に委譲）
2. **承認** — 分解結果をユーザに提示、承認 or 修正
3. **直交性判定** — 影響ファイル集合の交差 + 依存グラフ → グループ分け
4. **並行実行** — グループごとに Agent を並行起動（各 Agent は `/cycle` を実行）
5. **結果収集** — 各 Agent の成否を取得
6. **マージ** — 成功ブランチを順次マージ
7. **サマリー** — 統合レポート出力

分解のロジックは Agent に委譲するので、オーケストレータ自身はグルーコードのまま。

## ✅ Tests

このスキルはマークダウンベースの命令セットなので、従来のユニットテストは不要。
代わりに以下の検証シナリオで動作確認する：

### 分解 (Phase 0)
- [ ] 自然言語の複合指示 → 適切な粒度で複数 plan に分解される
- [ ] 依存関係が正しく検出される（A→B の順序）
- [ ] 独立した機能が正しく独立と判定される
- [ ] ユーザが edit を選択 → 修正指示を反映して再分解
- [ ] 分解結果が 0 plan → エラーメッセージで終了
- [ ] 分解結果が 1 plan → 通常の `/cycle` にフォールバック

### 直交性 & グループ化 (Phase 1)
- [ ] 影響ファイルが交差する plan → 別グループに分離
- [ ] 依存先が同グループ or 先行グループにいることの保証
- [ ] 完全独立な plan 群 → 全て同一グループ（1ラウンドで完了）
- [ ] 影響ファイルが空の plan → 任意のグループに配置される

### 並行実行 & マージ (Phase 2-3)
- [ ] 2つの独立した plan で並行実行 → 両方マージ成功
- [ ] 1つの cycle が失敗 → 他の cycle は成功・マージされる
- [ ] マージ後のテスト失敗 → 該当ブランチのみ revert
- [ ] 既存 plan ファイル指定 → Phase 0 スキップで Phase 1 から開始

## 🔒 Security

- [ ] worktree のクリーンアップ（成功・失敗問わず残骸を残さない）
- [ ] マージ前に必ず main の最新を pull（stale base 防止）
- [ ] force push は絶対にしない

## ⚠️ 既知の制限事項

1. **分解の品質は LLM 依存**: 指示が曖昧すぎると分解精度が下がる。ユーザ承認ステップで補正する想定。
2. **plan の「Files to Change」が正確である前提**: plan に書かれていないファイルを実装時に触った場合、直交性の保証が崩れる。ただし worktree 隔離があるので最悪でもマージ時に git が検出する。
3. **共有設定ファイルの扱い**: `package.json`, `tsconfig.json` などを複数 cycle が触る場合は並行不可になる。これは正しい挙動（安全側に倒れる）。
4. **テスト実行の前提**: マージ後のテスト実行はプロジェクトにテストランナーがある場合のみ。ない場合はスキップ。
5. **後続グループは先行グループの成果物に依存**: Group 1 で失敗した cycle があり、Group 2 がそれに依存している場合、Group 2 の該当 cycle もスキップされる。
6. **並行実行数の上限**: API レート制限を考慮し、同時実行する Agent は最大 3 に制限する。グループ内の cycle が 4 以上の場合はバッチ分割して直列に実行する。
7. **status.md への並行書き込み**: 複数 cycle が並行実行中は status.md への書き込みを抑止し、全 cycle 完了後にオーケストレータが一括更新する。session-history.md も同様。
