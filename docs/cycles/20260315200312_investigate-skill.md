## 📝 What & Why

LLMに不具合調査を依頼すると、問題を見つけた瞬間にそのまま修正を始めてしまう問題がある。調査結果に基づいて次のアクション（issue化・plan作成・議論継続など）をユーザーが判断できるよう、**調査専用**のスキルとコマンドを作る。

## 🎯 Goals

- 調査に特化し、**ファイル編集を絶対に行わない**スキルを作る
- 調査結果に影響範囲・規模感・確信度を含め、次のアクションを判断しやすくする
- 推奨アクションを選択肢として提示するが、勝手に実行しない

## 📐 Design

### Key Points

- **絶対不変の制約: ファイル編集禁止**: SKILL.md の冒頭に「Edit / Write / NotebookEdit ツールの使用を禁止」と明記。これがこのスキルの存在意義
- **調査フェーズの構造化**: 問題の再現確認 → 原因特定 → 影響範囲の分析 → 修正案の提示、という段階を踏む
- **推奨アクションの分類**: 調査結果の規模感に応じて、次に取るべきアクションの選択肢を提示する

### スキル設計

#### 入力
- ユーザーからの問題の説明（自然言語）
- 対象ファイルやエラーメッセージなど（任意）

#### 出力: 調査レポート
以下の構造で出力する（ファイルには書き出さず、会話上に表示。後から参照が必要な場合はユーザーが手動でコピーするか、`/issue-create` で記録する前提）:

1. **問題の概要** — 何が起きているか
2. **原因分析** — なぜ起きているか（根本原因）
3. **影響範囲** — どこまで影響するか
4. **確信度** — 調査結果の確からしさ（高/中/低 + 理由）
5. **修正案** — どう直すべきか（具体的なコード変更案を説明として示す。ただし実行はしない）
6. **推奨アクション** — 以下から適切なものを提案:

| 状況 | 推奨アクション | コマンド例 |
|------|---------------|-----------|
| 軽微な修正（1-2ファイル、明確な修正） | そのまま修正 or iterate | `/iterate` |
| スコープ外・後で対応 | issue として記録 | `/issue-create` |
| 中〜大規模な修正 | 計画を立ててサイクル | `/plan-create` → `/cycle` |
| 判断材料不足・要議論 | 追加調査や議論を継続 | （会話を続ける） |

#### 制約（SKILL.md に明記）

- **Edit ツール禁止**
- **Write ツール禁止**
- **NotebookEdit ツール禁止**
- 使えるツール: Read, Grep, Glob, Bash（読み取り専用コマンドのみ — ファイルシステムやリポジトリの状態を変更するコマンド（rm, mv, cp, chmod, git commit, git push 等）は禁止）, Agent（調査委譲のみ）, LSP

### コマンド設計

`commands/investigate.md`:
- 薄いラッパー。Skill ツール経由で `investigate` を呼び出すだけ
- 引数: 問題の説明（自由記述）
