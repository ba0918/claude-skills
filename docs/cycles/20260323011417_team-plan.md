# Team Plan スキル

**Cycle ID:** `20260323011417`
**Started:** 2026-03-23 01:14:17
**Status:** 🔵 In Progress

## 📝 What & Why

AgenticTeam 機能を使って、複数の専門家がチームで議論しながら実装計画を作成するスキル `team-plan` を新規作成する。

現在の `plan-create` は1エージェントが計画を作成するため、作成者の得意分野に偏った設計になりやすい。team-plan では最初から Security / Performance / Architect / Pragmatist の4人が議論に参加することで、初手から穴の少ない計画が生まれる。

team-cycle のレビューフェーズと同じチーム構成を使うが、目的が「レビュー（批判）」ではなく「共同作成（提案）」である点が異なる。

## 🎯 Goals

- チーム議論による多角的な計画作成の実現
- 既存の plan テンプレート互換の計画ファイル出力（docs/cycles/ 配下）
- skills/shared/ の team-config.md を team-plan と team-cycle で共有活用し、ロール定義の重複を避ける
- team-plan → team-cycle（または通常の cycle）の連携が自然にできること

## 📐 Design

### アーキテクチャ概要

```
team-plan コマンド
  │
  ├─ Phase 0: 準備
  │    ├─ Step 0.1: 環境変数チェック（CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS）
  │    │    └─ 未設定 → エラーメッセージを表示して中断
  │    └─ Step 0.2: ユーザーから要件を受け取り、コードベースを調査
  │
  ├─ Phase 1: チーム議論による計画作成（AgenticTeam、try-finally で TeamDelete 保証）
  │    │
  │    ├─ Step 1.1: TeamCreate（チーム名: team-plan-{timestamp}）
  │    │
  │    ├─ Step 1.2: Agent spawn × 4（並行、2名以上成功で続行、1名以下で中断）
  │    │    ├─ Security Advisor
  │    │    ├─ Performance Advisor
  │    │    ├─ Architect
  │    │    └─ Pragmatist
  │    │
  │    ├─ Step 1.3: 議論フロー
  │    │    ├─ Step 1: Lead が要件を全メンバーに共有
  │    │    ├─ Step 2: 各メンバーが設計提案・懸念を提出（SendMessage → Lead）
  │    │    ├─ Step 3: トレードオフについて議論（最大2ラウンド）
  │    │    └─ Step 4: Lead が合意に基づいて計画ファイル作成
  │    │
  │    └─ Step 1.4: TeamDelete（必ず実行 — 正常完了・エラーいずれの場合も）
  │
  ├─ Phase 2: 計画ファイル出力
  │    ├─ docs/cycles/{timestamp}_{slug}.md に生成（plan テンプレート準拠）
  │    └─ docs/status.md を更新（Skill `claude-skills:plan` 経由）
  │
  └─ 完了（計画ファイルパスを返す）
```

### Phase 0: 準備の詳細

#### Step 0.1: 環境変数チェック

```bash
echo "${CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS:-not_set}"
```

未設定（`not_set`）の場合:

```
⛔ TEAM-PLAN ABORTED: AgenticTeam feature not enabled

Set the environment variable before running:
  export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

Then retry: /claude-skills:team-plan
```

中断する。

#### Step 0.2: 要件受け取りとコードベース調査

1. `$ARGUMENTS` からユーザーの要件を取得（空なら質問）
2. Lead がコードベースを調査し、関連ファイルの構造・既存実装を把握する
3. 調査結果を整理し、Step 1.3 のメンバー共有に備える

### team-cycle との関係

```
skills/shared/references/team-config.md （共有ロール定義）
  │
  ├─ 参照 ← team-plan（計画作成）
  │           │ 独自: planning-flow.md（計画作成フロー）
  │           └─ 出力: 計画ファイル ─────────→ 入力
  │                                               │
  └─ 参照 ← team-cycle（レビュー+実装）            │
              │ 独自: review-flow.md（レビュー議論フロー）
              └─ 入力: 計画ファイル ←────────────────┘
```

- team-config.md は `skills/shared/references/` に移動し、team-plan と team-cycle の双方が参照する（横断依存を避ける）
- 各スキルは独自のフロー定義を持つ（planning-flow.md vs review-flow.md）

### チーム構成

team-cycle と同じ4ロールを使うが、**役割のフレーミングが異なる**:

| ロール | team-cycle（レビュー時） | team-plan（計画作成時） |
|--------|------------------------|----------------------|
| **Security** | 問題を指摘する批判者 | セキュリティ要件を提案する助言者 |
| **Performance** | 非効率を指摘する批判者 | 効率的な設計を提案する助言者 |
| **Architect** | 設計違反を指摘する批判者 | テスタブルな構造を提案する助言者 |
| **Pragmatist** | 過剰設計を指摘する批判者 | 実現可能なスコープを提案する助言者 |

> スポーンプロンプトで「レビュー」ではなく「設計提案」のフレーミングに切り替える。

### 計画作成の議論フロー

```
Step 1: 要件共有（Lead → 全員）
  Lead がユーザーの要件 + コードベースの関連情報を全メンバーに共有

Step 2: 設計提案（全員 → Lead、並行）
  各メンバーが自分の専門観点から:
  - 設計上考慮すべきポイントを提案
  - やるべきこと / やらないべきことを提案
  - 懸念事項があれば共有

Step 3: 議論（全員、最大2ラウンド）
  トレードオフが発生する論点について議論
  - 例: 「抽象化すべきか直接実装か」
  - 例: 「バリデーションをどの層でやるか」
  Lead が議論を収束に導く

Step 4: 計画作成（Lead）
  合意に基づいて計画ファイルを作成
  - plan テンプレートのフォーマットに準拠
  - 議論のハイライトを計画ファイルに含める
  - 各メンバーの提案がどう反映されたかを記録
```

### Key Points

- **team-config.md の共有**: ロール定義は `skills/shared/references/team-config.md` に配置。team-plan と team-cycle の双方が参照し、横断依存を回避する
- **チーム名の分離**: team-plan は `team-plan-{timestamp}` 、team-cycle は `team-review-{timestamp}` をチーム名に使う。同時実行時の衝突を防ぐ
- **チーム名の動的指定**: team-config.md からハードコードのチーム名セクションを削除し、ロールの専門知識定義に特化させる。チーム名は各スキルの SKILL.md 内で TeamCreate 時に動的に指定する
- **フレーミングの切り替え**: 同じロール定義を使いつつ、スポーンプロンプトの冒頭で「レビュー」→「設計提案」にフレーミングを切り替える。team-config.md 自体はロールの専門知識定義に特化し、タスクのフレーミングはスキル側で行う
- **コードベース調査の組み込み**: Step 1 で Lead がコードベースを調査し、関連ファイルの構造・既存実装を把握してからメンバーに共有する。メンバーにも Explore agent type の使用を許可し、自分で調べられるようにする
- **計画出力の互換性**: 出力する計画ファイルは `skills/plan/references/plan-template.md` に完全準拠。team-cycle でも通常の cycle でもそのまま使える
- **status.md の更新**: 計画作成後に Skill `claude-skills:plan` でステータスを更新する（Planning 状態として登録）
- **Phase 1 エラーハンドリング**: team-cycle と同じ try-finally パターンで TeamDelete を保証。spawn 失敗時は2名以上で続行、1名以下で中断
- **CLAUDE.md のアーキテクチャ更新**: `skills/shared/` を新レイヤーとして記載する。複数スキルが共有するリソース（ロール定義等）の配置場所として、既存の commands/skills の2層構造を補完する
