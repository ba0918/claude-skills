## 📝 What & Why

ユーザ指定範囲のコードを分析して問題を検出し、各問題を検索可能なパターンに変換してコードベース全体に横展開検索し、文脈検証（偽陽性除去）を挟んだうえで全該当箇所を修正する **find-one-fix-all 型スキル** を新規作成する。既存スキル（investigate=読み取り専用 / codebase-review=全体スコープ・修正なし / iterate=問題発見フェーズなし）が埋めていない「局所起点 → 横展開 → 一括修正」のワークフローを提供する。

## 🎯 Goals

- 指定範囲の分析 → 問題検出 → パターン化 → 全体横展開検索 → 文脈検証 → 修正までを一気通貫で実行するスキル `skills/sweep-fix/SKILL.md` を作成する
- **偽陽性除去を必須フェーズとして構造化する**（表面的に類似しているが文脈的に問題ないコードを修正対象から除外する。スキップ不可）
- 既存共有契約（severity-and-verdicts.md / orchestration-patterns.md / verification-gate.md）を再利用し、重複記述をしない
- skills-first 方針に従い command なし。description のトリガー語で起動導線を担保する
- `python3 scripts/validate_repo.py` 全チェック合格（ドキュメント対応表の更新を含む）

## 📐 Design

### ワークフロー設計（SKILL.md の中身）

```
Phase 0: SCOPE — 対象範囲の確定
  - 引数からファイル/ディレクトリ/glob/関数名をパース
  - 範囲未指定なら AskUserQuestion で確認（headless 時は git diff 起点にフォールバックしない。範囲必須で abort）
  - 範囲の存在確認（存在しないパスなら即エラー）

Phase 1: ANALYZE — 指定範囲の問題検出
  - 指定範囲を分析し問題リストを作成
  - 重大度は severity-and-verdicts.md の基準（Critical/High/Medium/Low）を流用
  - 問題ゼロなら「問題なし」レポートで正常終了（無理に指摘をひねり出さない）
  - 各問題に「修正案」と「一般化可能性」（横展開する価値があるか）を付記
  - 小規模範囲（〜10ファイル程度）はメインで直接実行、それ以上は単一 Agent 委譲
    （orchestration-patterns.md パターン1、model: opus — レビュー系はゲートがないため安くしない）

Phase 2: SWEEP — パターン化と横展開検索
  - 各問題を「検索可能なシグネチャ」に変換:
    (a) 字面パターン → Grep（正規表現）
    (b) 構文パターン → ast-grep（利用可能な場合。なければ Grep にフォールバック）
    (c) シンボル参照 → LSP（同一関数/型の全使用箇所）
  - 問題が複数ある場合は問題ごとに sweep エージェントを並行起動し、
    結果を .claude/tmp/sweep-fix/{n}_candidates.json に書かせてマージ
    （orchestration-patterns.md パターン2、model: opus、同一メッセージ内で並行発行）
  - 問題が1つなら並行化せず単一実行（過剰オーケストレーション回避）
  - 検索は読み取り専用。この段階では一切修正しない

Phase 3: VERIFY — 文脈検証（偽陽性除去）★このスキルの品質の要
  - 各候補箇所に対して references/context-verification.md のチェックリストで判定:
    - CONFIRMED   … 同じ問題が同じ理由で成立する → 修正対象
    - FALSE_POSITIVE … 字面は似ているが文脈的に問題ない → 除外（除外理由を記録）
    - UNCERTAIN   … 判断に必要な文脈が不足 → 修正せずレポートに掲載（ユーザ判断に委ねる)
  - 判定には根拠の明記を強制（「なぜこの箇所では問題が成立する/しないのか」）
  - UNCERTAIN を CONFIRMED に倒すことを禁止（fail-safe: 迷ったら直さない）

Phase 4: FIX — 修正適用
  - CONFIRMED 箇所のみ修正。Phase 1 の修正案を各箇所の文脈に合わせて適用
  - verification-gate.md 契約準拠: テストコマンドが判明していれば実行し、
    エビデンス（テスト出力・git diff）なしに完了を宣言しない
  - テスト失敗時: 自分の修正が原因なら修正、無関係な既存失敗なら区別して報告
  - コミットはしない（ユーザが /commit で行う。スキルの責務は修正まで）

Phase 5: REPORT — 構造化レポート
  - 検出した問題（重大度付き）/ 修正箇所一覧 / FALSE_POSITIVE 一覧（除外理由付き）/
    UNCERTAIN 一覧（ユーザへの判断材料付き）/ 検証エビデンス
  - .claude/tmp/sweep-fix/ の中間ファイルを削除
```

### Key Points

- **偽陽性対策が最重要**: Phase 3 を独立フェーズとして強制し、SKILL.md に合理化防止テーブルを入れる（「候補は明らかに同じ問題だから検証スキップ」→「字面の類似は文脈の同一を意味しない。例外なし」）。Phase 2 は「広めに検索して Phase 3 で絞る」方針とし、検索漏れ（偽陰性）は検索段階で、誤修正（偽陽性）は検証段階で、それぞれ担当を分ける
- **既存スキルとの棲み分けを description と SKILL.md 冒頭に明記**: investigate（調査のみ）/ codebase-review（全体レビュー・修正なし）/ iterate（ユーザが問題を持ち込む前提）との違い = 「局所で見つけた問題の横展開一括修正」
- **command なし**（skills-first）: description にトリガー語「sweep-fix」「横展開修正」「同様の問題を探して直して」「この範囲の問題を全部直して」等を含める。1024字以内
- **モデル階層**: Phase 1 分析 Agent・Phase 2 sweep エージェントは `model: opus` を明示（レビュー・発見系はゲートがないため安くしない / Fable セッションからの高額継承を防止）。attack-review ではないので fable 禁止条項は該当しないが、model 明示は必須
- **ast-grep / LSP は optional dependency**: 存在チェックしてなければ Grep フォールバック。外部ツールを可用性の前提にしない（orchestration-patterns.md パターン5 のフォールバック原則を検索ツールにも適用）
- **Codex 版は今回スコープ外**: 需要を見てから codex-sync で移植する（sync-manifest 更新も不要 = codex-skills/ に同名ディレクトリを作らない）
- **references は1階層まで**: SKILL.md → references/*.md で止め、参照チェーンを作らない（skill-authoring.md 準拠）
