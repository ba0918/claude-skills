## Summary

LLM 向け指示ファイル（CLAUDE.md / AGENTS.md / .claude/rules / プロジェクトメモリ）の老朽化・矛盾・有害指示・クロスツール乖離を監査する棚卸しスキル。長期運用による情報の腐敗を検出し、3値判定（AUTO_FIX / NEEDS_JUDGMENT / REPORT_ONLY）で低リスク修正のみ自動化する。v1 は純関数ルール中心の段階導入。

## Key Discussion Points

- **既存スキルとの差別化**: doc-check は docs⇔code、doc-audit は docs⇔docs の整合性検証。context-audit は「LLM の行動を左右する指示層の運用品質監査」として切る。doc-check は既に CLAUDE.md を検出対象に含むため、「記述がコードと合っているか」ではなく「指示としての品質（矛盾・有害性・死んだ制約・コンテキスト浪費）」を見ることが差別化の核
- **既存スキルが未カバーの領域**: (1) プロジェクトメモリ（`~/.claude/projects/*/memory/`、リポジトリ外で doc-check の射程外）(2) .claude/rules 等のユーザーレベル指示 (3) 「正しいけど有害」な指示の検出
- **検出4カテゴリ**: stale（参照切れ・git 履歴との乖離）/ contradiction（禁止と許可の衝突・優先順位不明）/ unsafe（破壊的操作の許可・確認省略・存在しないツール名）/ drift（Claude⇔Codex のクロスツール乖離）。design-lint の DL001 系に倣ったルール ID 体系（CA-S/C/U/D/M 系）
- **矛盾検出のスケール**: 全ペア比較は組み合わせ爆発するため claim 抽出パイプラインで解決 — 純関数で claim（`{subject, action, polarity, scope, source}`）を抽出 → subject key で bucket 化 → 同一 bucket 内で polarity が逆のペアのみ LLM 判定 → 「矛盾 / 意図的差分 / 優先順位で解決済み / 不明」の4分類。trigger-eval の「純関数は unittest、エージェントは JSON 受け渡しのみ」アーキテクチャを踏襲
- **baseline 台帳**: 意図的差分（CLAUDE.md と AGENTS.md のツール語彙差等）を台帳化して誤検知を抑制。sync-manifest の hash 方式を借りつつ、accepted difference に reason/subject を必須化、**expiry を導入**（意図的差分も老朽化する）、finding hash は正規化 claim から生成（行ズレ耐性）
- **メモリ監査の固有ルール例**: 一回限りの失敗談の feedback が恒久ルールとして強すぎる（CA-M403 相当）/ 特定環境の credential・path を他プロジェクトに適用しそうな user メモリ / 相対時制（「現在」「最新」）の劣化 / secret 疑いは自動マスクせず REPORT_ONLY
- **プライバシー**: メモリ監査のデフォルトは cwd 対応プロジェクトのメモリのみ。全プロジェクト横断は `--all-projects` に分離。グローバル設定（~/.claude/CLAUDE.md 等）はユーザーが明示指示した場合のみ
- **定期実行**: スキル自体は tick として作り、定期化は /loop・cron に委譲（issue-polling と同じ発想）。軽量な機械チェックのみ CI 組み込み可

## Decisions & Conclusions

- スキル名は **context-audit**（instruction-audit / context-inventory も候補だったが、指示ファイル + メモリを包含する広い名前を採用）
- v1 スコープ: **リポジトリ内指示ファイル + プロジェクトメモリ（cwd 対応分）**。グローバル設定は明示指示時のみ
- 検出軸は stale / contradiction / unsafe / drift の**4カテゴリ対等**のルール体系（+ メモリ専用 CA-M 系）
- 自動修正は **3値判定 + 低リスクのみ AUTO_FIX**（パス typo・実在ファイルへの置換・frontmatter 正規化程度）。削除は絶対に自動でやらない。sweep-fix / doc-audit の fail-safe パターン踏襲
- **v1/v2 の段階分け**: v1 は純関数ルール中心（stale 参照・schema・ツール語彙混入・カバレッジ差分）+ LLM 矛盾検出は REPORT_ONLY + baseline は単純 suppression リスト。claim 抽出パイプライン本格化と expiry 付き baseline 台帳は v2
- v1 の成功条件は「矛盾の完全検出」ではなく「claim 抽出と baseline が安定し、次回以降の差分監査が安くなること」
- skills-first 方針により command なしで作る

- 出力フォーマットの詳細（investigate 型レポート？ doc-audit 型の修復ログ？）

- skill-improve との接続（実際の失敗セッションから「指示ファイル起因の摩擦」を検出する導線）
- 発展モード案の優先度: `--explain-load-order`（有効指示スタック可視化）/ `--claims`（claim JSON ダンプ）/ `--prune`（削除候補提案）/ `--risk`（危険指示のみ抽出）
